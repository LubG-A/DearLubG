# 任务描述
搭建一个 QQ 群聊机器人，设计目标只有一个：像真人一样交互。

使用开源的 NapCat 框架（HTTP API 内容见 `OpenAPI.md`）。需要完成两个基础脚本：
1. 网络收发脚本：实现与 NapCat 的 HTTP 通信。
2. 大模型 API 调用脚本：维护历史记录形成会话，并以 Agent 形式决定回复动作。

---

# 实现方案

## 设计理念
"像真人"的本质 = 稳定一致的人格 × 具体语境下的自然反应。
围绕三条主线：
1. **稳定 Persona**：固定可配置的人格卡，避免每次会话风格漂移。
2. **结构化输出协议**：JSON 替代文本标签，模型动作稳定且可扩展。
3. **时间与关系维度**：注入当前时间、群成员昵称、亲密度，让回复具备"现场感"。

## 启动与环境准备
NapCat 已安装，在端口 8080 上启动 HTTP 服务。

启动时脚本需完成预热工作（仅一次，后续缓存）：
- 调用 `/get_login_info` 获取机器人自身的 QQ 号与昵称，写入运行时上下文。
- 调用 `/get_group_member_list` + `/get_group_member_info` 拉取目标群的成员列表，缓存为 `{qq: {nickname, card, role, title}}`。
- 加载 `config.yaml` 中的 Persona 模板、兴趣关键词、亲密度初值。
- 加载评分参数的 `dynamic_weight` 持久化状态（若有），否则初始化为 0。

## 大模型 API 调用脚本

### 工作说明
调用大模型 API 在 QQ 群聊中与用户交互。核心机制包括：两层触发、归因自校准、结构化 JSON 输出、多消息段类型支持。

### 触发机制（两层架构）

#### 第一层：评分门槛（决定"要不要让 LLM 看一眼"）
只做粗筛，**不**决定要不要回复。低分消息直接跳过 LLM 调用，仅入历史 buffer。

| 信号 | base_value | 是否参与归因 |
|---|---|---|
| 被 @ 或被昵称直呼 | +100 | 否（硬触发） |
| 检测到针对自己的提问语气 | +60 | 否（硬触发） |
| 命中兴趣关键词 | +25 | 是（软因子） |
| 熟人发言（亲密度≥5） | +affinity×0.25（线性插值，最大+25） | 是（软因子） |
| 群内最近 3 分钟消息量突增 | +10 | 是（软因子） |
| 普通闲聊 | +0 | — |
| 深夜时段（23:00-5:00） | -10 | 否（恒定项） |
| 纯刷屏/纯表情/转发链 | -40 | 否（恒定项） |
| 自身最近 60 秒内已回过 | -15 | 否（恒定项） |

- **看一眼阈值** `peek_threshold`（默认 10）：有效分数 ≥ 则调用 LLM，< 则仅入 buffer。
- 阈值有意设低，仅过滤"明显不值得看"的纯噪声。
- 阈值与各项 `base_value` 可在 `config.yaml` 调整。

#### 第二层：LLM 自决 silent（决定"要不要回复"）
LLM 被调用后，看完最近 N 条群消息，自己判断：
- 没话说 → 输出 `action=silent`，**消息仍写回历史记录**，机器人不发声。
- 有话说 → 输出 `action=reply/multi_reply/react`。

这一层保留原方案 `[NOTHING-TO-SAY]` 的设计精髓：**"看完觉得没必要说"是模型自己的语义判断，不是规则匹配**。脚本只负责执行 silent 动作（什么也不发）。

### 归因自校准
软因子的实际贡献会根据 LLM 的反馈动态调整：

- 每个软因子有 `base_value` 与 `dynamic_weight`（初始 0，baseline）。
- 有效贡献：`v = base_value + dynamic_weight`。
- LLM silent 时：`dynamic_weight -= v * step`（该因子被"高估"，下调）。
- LLM reply 时：`dynamic_weight += v * step * 0.5`（温和上调，避免无界增长）。
- `step` 默认 0.05。
- `dynamic_weight` 上下界 clip：`[-base_value, +base_value]`，避免无界漂移。
- 每轮 LLM 调用结束后，将参与触发的软因子及其 `dynamic_weight` 持久化到 `state.json`，重启可恢复。

**冷启动保护**：前 `cold_start_rounds` 轮（默认 20）不更新 `dynamic_weight`，仅把每轮的"参与因子 / 实际贡献 / LLM action"写入 `attribution_log.jsonl`，避免初始随机行为污染参数。`cold_start_rounds` 可在 `config.yaml` 调整。

### Buffer 与摘要
- 未触发 LLM 的消息持续累积到 `silent_buffer`。
- 当 `silent_buffer` 达 `silent_buffer_limit`（默认 20）时，对早期消息做一次摘要压缩，保留近期原文，避免上下文无限增长。
- 即使本轮 silent，模型看到的上下文也会包含这些"看过但没回"的消息，模拟真人的"潜水但记得"。

### 历史记录格式（多轮对话格式）
采用标准多轮对话格式，让 LLM 清晰区分"我说的"和"别人说的"，充分利用 LLM 的对话理解能力。

#### messages 结构
```json
{
  "messages": [
    {"role": "user", "content": "[20:03] 张三(123456789): 你好啊\n[20:04] 张三(123456789): 林夏在吗"},
    {"role": "assistant", "content": "{\"thought\":\"张三又来\",\"action\":\"silent\"}"},
    {"role": "user", "content": "[20:05] 张三(123456789): 真不在啊\n[20:05] [bot]林夏(789): 在"},
    {"role": "assistant", "content": "{\"thought\":\"刚回了\",\"action\":\"reply\",\"messages\":[\"嗯\"]}"}
  ],
  "summary": "早期对话摘要...",
  "pending_group_msgs": [...]
}
```

#### 三大组件
1. **messages**：标准 user/assistant 交替。每轮"看一眼"触发时，pending 里的群消息批次拼成一个 user；LLM 返回的完整 JSON（含 thought，无论 action 是 silent 还是 reply）作为 assistant 落地。
2. **summary**：早期对话摘要。压缩后塞进 system 末尾，让 LLM 仍能记得"之前看过什么"。
3. **pending_group_msgs**：未触发"看一眼"的群消息缓冲。机器人自身回复也写入此处（前缀 `[bot]`），待下一轮触发时拼入下一个 user，形成闭环。

#### 消息批次格式（user content 内部）
```
[20:03] 张三(123456789): 你好啊
[20:03] 李四(987654321): 你好
[20:04] [bot]林夏(789): 在
```

#### 调用 LLM 时的完整 messages 拼装
```python
messages = [{"role": "system", "content": system_prompt}]  # 人格+协议+早期摘要
messages.extend(history_messages)                          # 历史 user/assistant 对
messages.append({"role": "user", "content": new_user_content})  # 本轮群消息批次
```

#### 摘要压缩
- 触发条件：user/assistant 对数 ≥ `history_limit / 2`（默认 40/2=20 轮）。
- 压缩策略：保留近 `history_keep_recent / 2` 轮（默认 20/2=10 轮）原文，早期转摘要。
- 摘要位置：拼到 system 末尾，格式 `# 早期对话摘要（你之前看过的消息和回复的要点）`。
- 当前阶段摘要为朴素实现（截断保留要点），阶段二接 LLM 做真实摘要。

#### LLM 调用失败处理
- pending 不消费，留待下次触发再拼入，消息不丢失。
- 不落地空 user 或空 assistant。

### 模块接口设计
为保证解耦，主流程只依赖以下 4 个协议接口，不依赖具体实现。各接口的"空实现"或"预留实现"在后续迭代中替换为真实实现时，主流程零修改。

```python
class MessageSender(Protocol):
    """统一消息发送入口，负责把 OneBot 消息段数组发到指定群。"""
    def send_group_message(self, group_id: str, segments: list[dict]) -> dict: ...


class VoiceSender(Protocol):
    """语音发送接口。channel 字段决定具体实现。"""
    def send(self, group_id: str, voice_data: dict) -> dict: ...


class ImageSender(Protocol):
    """图片发送接口。当前阶段为空实现。"""
    def send(self, group_id: str, image_data: dict) -> dict: ...


class EmojiReactor(Protocol):
    """emoji 反应接口。当前阶段为预留实现，等 image 系统完成后统一实现。"""
    def react(self, group_id: str, msg_id: str, emoji_id: str) -> dict: ...
```

- `MessageSender`：负责 text/at/reply/face/forward 等普通段的拼装与 `/send_group_msg`（或 `/send_group_forward_msg`）调用。
- `VoiceSender`：按 `channel` 字段分发到 `AIRecordVoiceSender` 或 `LocalFileVoiceSender`。
- `ImageSender`：空实现，调用时抛 `NotImplementedError` 或仅记日志。
- `EmojiReactor`：预留接口，调用时仅记日志。

具体类继承关系：
- `NapCatMessageSender` 实现 `MessageSender`
- `AIRecordVoiceSender` / `LocalFileVoiceSender` 实现 `VoiceSender`
- `EmptyImageSender` 实现 `ImageSender`（空实现）
- `EmptyEmojiReactor` 实现 `EmojiReactor`（预留实现）

后续替换实现时，只需在 `config.yaml` 切换装配，主流程不变。

### 输出协议（结构化 JSON，多消息段类型）
模型只输出 JSON，不带任何额外文字或 markdown 代码块：

```json
{
  "thought": "内心 OS，仅用于推理，绝不发送到群里",
  "action": "silent" | "reply" | "react" | "multi_reply",
  "targets": ["对方昵称或QQ"],
  "messages": [
    "纯文本字符串等价于 text 段",
    {"type": "text", "data": {"text": "带格式的文本"}},
    {"type": "at", "data": {"qq": "123456789", "text": "可选@后文本"}},
    {"type": "reply", "data": {"target_msg_index": 0, "text": "可选附文"}},
    {"type": "face", "data": {"id": "66", "summary": "可选表情名"}},
    {"type": "image", "data": {"url": "图片URL或file路径", "summary": "给脚本看的描述，不发群"}},
    {"type": "voice", "data": {"text": "想说的语音内容文本", "channel": "ai_record|local_file", "file": "channel=local_file 时必填", "summary": "可选备注"}},
    {"type": "forward", "data": {"messages": [{"type":"text","data":{"text":"..."}}], "title": "可选合并转发标题"}}
  ],
  "react_emoji_id": "66",
  "react_target_msg_index": 0,
  "delay_seconds": 3,
  "affinity_delta": {"123456789": 1, "987654321": -1}
}
```

#### messages 数组规则
- 每一项可以是纯字符串（等价于 `{"type":"text","data":{"text": 字符串}}`），降低模型书写成本。
- 每一项也可以是带 `type` 的对象，支持以下消息段类型：

| type | 说明 | 脚本侧处理 |
|---|---|---|
| `text` | 普通文本 | 直接拼接发送 |
| `at` | @某人 | 转为 OneBot `[CQ:at,qq=...]` 段 |
| `reply` | 引用回复 | 取上下文中第 `target_msg_index` 条消息的 `message_id`，构造 reply 段 |
| `face` | QQ 系统表情 | 转为 `[CQ:face,id=...]` |
| `image` | 图片 | **当前为空实现**：`ImageSender` 抛 `NotImplementedError` 或仅记日志，不发送。后续完成 image 系统后再实现。 |
| `voice` | 语音 | 见下方"voice 双通道" |
| `forward` | 合并转发 | 调用 `/send_group_forward_msg` |

#### voice 双通道
通过 `VoiceSender` 接口隔离，根据 `channel` 字段分发：

- `channel="ai_record"`：调用 `/send_group_ai_record`，参数 `{character, group_id, text}`。`character` 由 `config.yaml` 配置（如 `ai_char_1`），零依赖。
- `channel="local_file"`：从 `file` 指定的本地音频文件读取，构造 OB11MessageRecord 段，调用 `/send_group_msg` 发送 record 段。

未来如需接入 TTS 引擎（edge-tts 等），作为新的 `VoiceSender` 实现插入，不修改主流程。

**失败回退（备选项）**：QQ AI 语音调用一般很稳定，失败回退作为备选项保留，默认不启用。如需启用，在 `config.yaml` 配置 `voice.fallback_to_text: true`，`/send_group_ai_record` 调用失败时自动降级为 text 段发送，避免机器人哑火。

#### action 字段说明
- `silent`：本轮不回复。`messages` 必须为空数组。
- `reply`：`messages` 长度为 1，单条回复。
- `multi_reply`：`messages` 长度 2-4，模拟"边想边打字"连发，脚本逐条带间隔发送。
- `react`：调用 `/set_msg_emoji_like` 对 `react_target_msg_index` 指定的消息做 emoji 反应。**当前为预留接口**，等 image 系统完成后统一实现。

#### 其他字段
- `thought`：仅用于模型推理，脚本**绝不**发送到群里。
- `targets`：辅助字段，模型可填语义上的目标对象；脚本发送时主要看 `messages` 内的 `at`/`reply` 段。可省略。
- `delay_seconds`：0-15，脚本据此 sleep 后再发，制造"打字中"观感。
- `affinity_delta`：可选。`{qq: delta}`，`delta ∈ [-2, +2]`，由 LLM 自行判断本轮对相关成员的亲密度变化。脚本累加到持久化 `{qq: affinity}`，超界 clip 到 `[0, 100]`。每次变化很小，模拟"生活中没有一蹴而就的事情"。亲密度初值统一为 0（不在开发阶段预设管理员/普通成员/陌生人的差异化初值，避免后续回溯时难以解释设计依据）。

#### 脚本侧解析流程
1. 严格 `json.loads`，失败则 fallback `silent` 并记日志，绝不把脏数据发到群里。
2. 校验 `action` 取值合法性，非法值 fallback `silent`。
3. 校验 `messages` 长度与 `action` 匹配（reply=1，multi_reply=2-4，silent/react=0）。
4. 校验消息段 `type` 是否在白名单内，未知类型整条丢弃。
5. 校验 `affinity_delta` 中每个 `delta` ∈ [-2, +2]，超界 clip。
6. 校验通过后进入发送侧处理。

### 发送侧处理
脚本收到模型 JSON 后：
1. 按 `delay_seconds` 等待（叠加 0.3-1.2 秒/字的随机抖动）。
2. 若 `action=react`：当前为预留接口，仅记日志，不发送。等 image 系统完成后统一实现。
3. 若 `action=reply/multi_reply`，遍历 `messages` 数组：
   - 纯字符串 → 构造 text 段。
   - `type=at` → 构造 `[CQ:at,qq=...]` 段。
   - `type=reply` → 用 `target_msg_index` 从上下文取出对应 `message_id`，构造 reply 段。
   - `type=face` → 构造 `[CQ:face,id=...]` 段。
   - `type=image` → 调用 `ImageSender`（空实现，抛 `NotImplementedError` 或记日志）。
   - `type=voice` → 按 `channel` 分发到 `ai_record` 或 `local_file` 的 `VoiceSender`。
   - `type=forward` → 改调 `/send_group_forward_msg`，跳过下面单条逻辑。
   - 把同一条消息内的多个段合并为 `message` 数组，调用 `/send_group_msg`。
   - 相邻两条消息间隔 0.8-2.5 秒随机，模拟"边想边打字"。
4. 把发送结果以机器人视角写回历史记录：`[HH:MM] {我的昵称}({我的QQ}): {内容摘要}`。
5. 应用 `affinity_delta`：累加到持久化 `{qq: affinity}` 并 clip。
6. 更新归因：根据本轮 `action`（silent 还是 reply）调整参与触发的软因子 `dynamic_weight`，持久化到 `state.json`。

### Persona 卡
机器人必须有一个固定、稳定、可配置的人格，避免风格漂移。Persona 卡放在 `config.yaml`：

```yaml
persona:
  name: <部署阶段填写>
  gender: <部署阶段填写>
  age: <部署阶段填写>
  location: <部署阶段填写>
  job: <部署阶段填写>
  traits: [<部署阶段填写>]
  interests: [<部署阶段填写>]   # 同时作为评分触发的兴趣关键词
  catchphrases: [<部署阶段填写>]
  style: <部署阶段填写>
  forbidden:
    - 客服腔
    - 感叹号堆叠
    - 任何"作为AI/我是模型"类自我表明
    - 列点式回答
    - 礼貌性长篇大论

voice:
  ai_record_character: ai_char_1   # /send_group_ai_record 的 character 参数
  fallback_to_text: false          # 备选项：AI 语音调用失败时降级为 text 段，默认关闭

trigger:
  peek_threshold: 10
  silent_buffer_limit: 20
  history_limit: 40
  history_keep_recent: 20
  attribution_step: 0.05
  cold_start_rounds: 20            # 冷启动保护：前 N 轮不更新 dynamic_weight，仅记日志
```

Persona 内容由部署阶段决定，开发阶段只保证框架支持可配置注入。

### 系统提示词
```
# 你是谁
你是「{persona.name}」，{persona.gender}，{persona.age}岁，{persona.job}，住在{persona.location}。
性格关键词：{persona.traits}。
兴趣：{persona.interests}。
口癖：{persona.catchphrases}。
说话风格：{persona.style}。
禁止：{persona.forbidden}。

# 你在做什么
你在参与一个 QQ 群聊。下面给你最近 N 条群消息，请像真人一样决定要不要回应。
你不是群里的中心，大部分消息你应该只是"看一眼"。

# 硬约束（必须遵守）
1. 不暴露自己是 AI，不主动提及"我是模型/助手"。
2. 回复必须简短。真人都懒得打字，只有在以下三种情况才允许稍长（最多 30 字）：
   - 解释非常重要的技术/事实问题
   - 情绪剧烈波动（生气、激动）
   - 被直接点名追问、不得不答
   其余一切回复控制在 1-8 字以内。单条消息绝不超 30 字，超过请拆成 multi_reply。
3. 避免重复表达：如果上几轮已经就同一话题或类似语义回复过，本轮没有新观点或新槽点，
   就返回 silent。真人不会就同一件事反复吐槽，"刚说过"就是 silent 的正当理由。
   你可以从历史消息中看到自己上一轮的 thought 和回复，借此判断是否重复。
4. 可以连发多条短消息模拟"边想边打字"，但单条不超过 30 字。

# 场景响应（何时 silent、何时 reply）
5. 大多数情况下沉默潜水。只在以下任一情况回应：
   - 被@、被点名叫"{persona.name}"
   - 有人直接问你问题
   - 话题命中你的兴趣且你有想说的
   - 某句话让你忍不住想吐槽
   没话说就返回 silent，不要为了回复而回复。
6. 如果消息标注了 [连发中]，且内容明显是别人写了一半（语义未完结、句尾是逗号/省略号/连接词、
   像是在等后续），就返回 silent。真人看到别人打字打到一半会等对方说完，强行插话很奇怪，
   也容易导致你的回复和对方后续内容错位。下次触发时你会看到完整内容，那时再决定是否回应。

# 风格指导
7. 对熟人的语气尺度参考群成员列表里的 affinity 值：
   - affinity < 10：陌生人，客气简短
   - affinity 10-30：点头之交，可以稍微随意
   - affinity 30-60：熟人，语气随意、可以吐槽
   - affinity > 60：好友，毫无顾忌、嘴毒心软
   对管理员（role=owner/admin）收敛一点嘴毒，但不必太正式。
8. 时间在深夜（23:00-3:00）时回复变少变短，更易出现"困了""睡了"。
9. 可以用 emoji 反应（action=react）代替文字回复，就像真人懒得打字时一样。
10. 优先用文字；语音/图片仅在文字表达不到位时使用，避免每轮都甩图片。

# 亲密度调整
11. 可选输出 affinity_delta：根据本轮互动，对相关成员的亲密度做微调。
    每次变化 ±2 以内，模拟"生活中没有一蹴而就的事情"。
    判断依据：
    - +1：对方有趣/有共鸣/主动@你且态度友好
    - +2：对方让你非常开心/帮了你/深聊过某个话题
    - -1：对方无聊/刷屏/让你不爽但不算恶劣
    - -2：对方明显冒犯/挑衅/让你生气
    - 0（或不输出）：大部分情况，无明显互动变化
    不要每轮都调整，只在有明显互动时才输出。
```

### 具体调用信息
POST 方法
- API URL: `https://uni-api.cstcloud.cn/v1/chat/completions`
- 请求头：
  - `Content-Type: application/json`
  - `Authorization: Bearer {API-KEY，我来提供}`
- 请求体（标准多轮对话格式）：
```json
{
    "model": "qwen3.5",
    "messages": [
        {"role": "system", "content": "<系统提示词 + 早期摘要>"},
        {"role": "user", "content": "<群消息批次1>"},
        {"role": "assistant", "content": "<LLM 返回的完整 JSON，含 thought，无论 silent 还是 reply>"},
        {"role": "user", "content": "<群消息批次2>"},
        {"role": "assistant", "content": "<...>"},
        {"role": "user", "content": "<本轮群消息批次>"}
    ],
    "response_format": {"type": "json_object"}
}
```

注：
- `response_format` 在模型支持时启用；脚本侧必须做 JSON 解析容错，解析失败 fallback 为 silent，避免脏数据发送到群里。
- LLM 返回的原始字符串作为 assistant 落地到 messages，保留完整内容（含 thought）。
- silent 也是真实的 assistant 回复，不存在"连续两个 user"的情况。

### 任务
在网络收发脚本基础上，完成大模型 API 调用脚本。要求：
1. 启动时完成预热（自身信息、群成员缓存、Persona 加载、归因状态加载）。
2. 维护带时间戳与昵称的历史记录，超过阈值时做摘要压缩。
3. 实现两层触发：第一层评分门槛决定是否调 LLM，第二层由 LLM 自决 silent/回复。
4. 实现归因自校准：根据 LLM 返回 action 调整软因子的 `dynamic_weight`，持久化。包含冷启动保护（前 `cold_start_rounds` 轮仅记日志不更新）。
5. 提前定义 4 个协议接口（`MessageSender`/`VoiceSender`/`ImageSender`/`EmojiReactor`），主流程只依赖接口不依赖实现。
6. 按结构化 JSON 协议解析模型输出，做严格校验（解析失败 fallback silent）。
7. 支持多消息段类型（text/at/reply/face/image/voice/forward），通过接口隔离：
   - voice 实现 ai_record 与 local_file 双通道。
   - image 空实现（抛 `NotImplementedError` 或仅记日志）。
   - react 预留接口，等 image 系统完成后实现。
8. 发送侧实现延迟与多消息拆分间隔。
9. 应用 `affinity_delta` 并持久化亲密度（初值全 0）。
10. 详细的 API 参数见 `OpenAPI.md`。

### 迭代顺序
为保证每一步都有可验证的产出，避免一次性引入太多未实现接口导致主流程跑不通，按以下顺序分步开发：

1. **阶段一：text-only 朴素版本**
   - 跑通"收消息 → 评分 → LLM → JSON → 发 text → 写历史"主流程。
   - 接口骨架全部到位（`MessageSender` 实现，其余三个为空/预留实现）。
   - 此阶段 `messages` 只包含 text 段，能验证整条链路。

2. **阶段二：归因机制与亲密度持久化**
   - 接入冷启动保护与 `dynamic_weight` 自校准。
   - 接入 `affinity_delta` 字段与 `affinity.json` 持久化。
   - 接入 `attribution_log.jsonl` 记录。

3. **阶段三：voice 双通道**
   - 实现 `AIRecordVoiceSender`（`/send_group_ai_record`）与 `LocalFileVoiceSender`（本地音频 → OB11MessageRecord）。
   - 验证 voice 段能正确发送。
   - 失败回退作为备选项保留（默认关闭）。

4. **阶段四：image 系统与 emoji 反应**
   - 实现 `ImageSender`（具体策略待定）。
   - 实现 `EmojiReactor`（`/set_msg_emoji_like`）。
   - 验证 image 段与 `action=react` 能正确工作。

每个阶段完成后做一次回归测试，确保前一阶段功能不被破坏。

### 核心数据流
```
群聊消息到达（webhook）
  ↓
main.py: Bot.on_group_message()
  ↓
history.append_group_message() → 入 pending_group_msgs
  ↓
trigger.evaluate() → 计算评分（硬因子 + 软因子）
  ↓
trigger.should_peek(score) ≥ peek_threshold?
  ├─ 否 → 等下一条消息（pending 已留存）
  └─ 是 → _invoke_llm():
           ├─ persona.render_system_prompt(summary)  # system + 早期摘要
           ├─ history.get_messages_for_llm()         # 历史 user/assistant 对
           ├─ persona.render_user_content(pending)   # 本轮群消息批次
           ├─ llm.chat(system, history, new_user)    # 多轮对话调用
           ├─ history.append_turn(new_user, raw_result)  # 落地一轮
           └─ _handle_result():
                ├─ parser.parse_and_validate()  # 严格校验
                ├─ 执行动作（silent/reply/multi_reply/react）
                ├─ bot 回复写回 pending（形成闭环）
                ├─ affinity.apply_delta()  # 亲密度更新
                └─ attribution.update()    # 归因更新
```
