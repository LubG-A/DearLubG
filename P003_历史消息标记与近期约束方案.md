# P003 方案 A+B：历史消息标记与近期约束

## 背景

P002 问题（长上下文下 LLM 答非所问）在 v1.4 通过对摘要添加约束文本得到部分缓解，但残留问题：Bot 仍会偶尔对历史对话中很久之前的消息进行回复。

### 残留问题的根因

当前历史消息在送给 LLM 时，**每条历史 user message 都是格式完整的"独立请求"**：

```
[user] # 当前上下文
       - 当前时间：2026-07-09 15:23 Thursday
       - 你的昵称：LubG（QQ：3952966724）
       - 状态：被动触发...
       - 群成员（昵称/QQ/角色/亲密度）：
       忆雨​(3184546563) role=member affinity=3
       ...（7行成员列表）
       # 最近群消息
       [#2013481826] [15:18:12] 忆雨​(3184546563): 我没意见
       ...
       请输出 JSON。
[assistant] {"action": "silent"}
```

从 LLM 视角的问题：
1. 每条历史 user message 末尾的"请输出 JSON"是一个**强触发信号**，让 LLM 觉得"每个位置都是可以回复的点"
2. 历史与当前请求**格式完全相同**，LLM 在长上下文中注意力漂移时无法区分"已处理过的历史"和"需要回复的当前请求"
3. 历史中那些当时 silent 的轮次，LLM 可能觉得"上次没回，这次可以补一下"

---

## 方案 A：历史消息标记 + 删除"请输出 JSON"

### 核心思路

在 `get_messages_for_llm()` 返回前，对历史消息（除最后一条 user）做转换：
- user content 开头的 `# 当前上下文` 替换为 `# 历史上下文`
- user content 末尾的 `请输出 JSON。` 删除

### 改动位置

**仅改 `src/history.py` 的 `get_messages_for_llm()` 方法**，不改存储格式（conversation.json 仍保留原样），只在送给 LLM 前做转换。

这样保证：
- 存储可逆，不影响其他功能（如 `get_recent_speakers` 扫描原 content）
- 改动集中在单一方法，易于验证和回滚

### 实现细节

```python
def get_messages_for_llm(self) -> list[dict]:
    """返回传给 LLM 的 messages 刪表（不含 system）。
    
    对历史消息做标记转换：
    - 除最后一条 user 外，user content 的 "# 当前上下文" → "# 历史上下文"
    - 除最后一条 user 外，删除末尾的 "请输出 JSON。"
    目的：消除历史中的"回复触发点"，让 LLM 明确区分历史与当前请求。
    """
    messages = self.messages.copy()
    if not messages:
        return messages
    
    # 找到最后一条 user 的索引
    last_user_idx = None
    for i in range(len(messages) - 1, -1, -1):
        if messages[i]["role"] == "user":
            last_user_idx = i
            break
    
    # 对除最后一条 user 外的所有 user content 做转换
    result = []
    for i, m in enumerate(messages):
        if m["role"] == "user" and i != last_user_idx:
            content = m["content"]
            # 替换开头标记
            if content.startswith("# 当前上下文"):
                content = "# 历史上下文" + content[len("# 当前上下文"):]
            # 删除末尾的"请输出 JSON。"
            if content.endswith("\n\n请输出 JSON。"):
                content = content[:-len("\n\n请输出 JSON。")]
            elif content.endswith("\n请输出 JSON。"):
                content = content[:-len("\n请输出 JSON。")]
            elif content.endswith("请输出 JSON。"):
                content = content[:-len("请输出 JSON。")]
            result.append({"role": "user", "content": content})
        else:
            result.append(m)
    return result
```

### 关键考虑

1. **只改 user，不改 assistant**：历史 assistant 回复是过去的事实，保持不变
2. **只改除最后一条外的 user**：当前请求必须保留完整的"# 当前上下文"和"请输出 JSON"
3. **转换在返回时做，不持久化**：conversation.json 仍存储原始格式，避免影响其他功能
4. **正则/字符串匹配的鲁棒性**：用 endswith 三段匹配（带前置换行符的变体），应对历史上可能的格式微调

### 预期效果

LLM 看到的消息结构变为：
```
[user] # 历史上下文
       - 当前时间：...（元信息保留，不影响理解）
       ...
       # 最近群消息
       ...（无"请输出 JSON"）
[assistant] {"action": "silent"}
...
[user] # 当前上下文       ← 只有当前请求保留完整格式
       ...
       请输出 JSON。
```

历史中的"回复触发点"被移除，LLM 注意力自然聚焦到唯一保留"请输出 JSON"的当前请求。

### 前缀缓存影响

**轻微**。历史部分文本变化（开头几字 + 末尾几字），但 messages 数组结构不变（仍是 user/assistant 交替）。前缀缓存从 system_prompt + 第一条 user 开始匹配，第一条 user 的开头从"# 当前上下文"变成"# 历史上下文"会导致该条之后缓存失效——但这部分本来就是每次变化的（因为是历史，跨轮次会累积）。

实际影响：与当前相比，缓存命中范围基本不变（system prompt 部分仍稳定）。

## 方案 B：persona 约束近期回复

### 核心思路

在 persona 系统提示词中，明确告知 LLM 主要针对最近的几条消息回复，不要主动提及遥远的历史内容。

### 改动位置

**`src/persona.py` 的 `SYSTEM_PROMPT_TEMPLATE`**，在"# 场景响应"或"# 理解上下文与指代"节中添加一条新规则。

### 实现细节

在"# 理解上下文与指代"节末尾追加一条规则：

```
- 历史对话中标记为"# 历史上下文"的内容是已经处理过的旧消息，仅供你理解上下文延续性，
  不要主动对这些旧消息进行回复或补充。你的回复对象应当是当前请求（"# 当前上下文"）
  中的最近群消息。只有在当前话题与历史内容存在自然关联时，才可以提及历史信息作为补充。
```

### 关键考虑

1. **单独效果有限**：v1.4 的经验证明，persona 约束在长上下文（2-3万 token）后会被严重稀释。方案 B 是"辅助"而非"主力"
2. **与方案 A 协同**：方案 A 在格式上消除歧义，方案 B 在语义层面补充约束。A+B 一起做的效果远大于单独做
3. **措辞要避免绝对化**：不写"绝不回复历史"，因为有时当前话题确实需要引用历史（如"刚才谁说的XX"）。用"不要主动"和"除非自然关联"留出空间

### 预期效果

在方案 A 已消除格式歧义的基础上，进一步在语义层面告诉 LLM"历史是参考材料"。当 LLM 注意力漂移到历史时，persona 约束提供"不应该回复"的信号。

## 验证方案

### 基础验证

1. 运行 `debug_reproduce.py` 脚本，用相同的 conversation.json 输入，看复现率是否下降
2. 对比改动前后的 LLM 返回（temperature=0.0，多次调用）：
   - 改动前：4次有效结果完全不一致，全部回复旧话题
   - 改动后预期：多数结果应正确回复当前消息或 silent

### 上线观察

1. 部署后观察 1-2 天，看"答非所问"频率是否下降
2. 关注 P002 中记录的"回复历史旧话题"现象是否消失或显著减少
3. 注意是否有误伤：当前话题确实延续历史时，bot 是否还能正常接话

### 回滚预案

- 方案 A：`get_messages_for_llm()` 恢复为 `return self.messages.copy()`，一行回滚
- 方案 B：删除 persona 中新增的规则文本，一行回滚
- 两个方案互相独立，可以单独回滚

## 未覆盖的问题

方案 A+B 解决的是"历史格式歧义导致注意力漂移"。但不解决：
- 历史原文过长导致的注意力稀释（需要重构方案）
- 历史中冗余元信息（成员列表等）占据 token（需要重构方案）
- 摘要锚定效应（已被 v1.4 部分缓解）
