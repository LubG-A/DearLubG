# P004 方案：历史消息合并与近期对话聚焦（重构）

## 背景

P003（方案 A+B）通过标记历史和 persona 约束，可以缓解"历史格式歧义导致的注意力漂移"。但残留问题：

1. **历史原文过长**：60 轮历史原文（约 3 万 token）导致注意力稀释（Lost in the Middle），末尾当前消息权重不足
2. **冗余元信息**：每条历史 user message 都带完整元信息（群成员列表、时间、状态），在历史中完全无用却占据大量 token 和注意力
3. **历史与当前格式相同**：即使 P003 标记了"# 历史上下文"，历史消息的结构仍然与当前请求类似，LLM 仍可能混淆

本方案是 P003 的进阶重构，**从结构上彻底分离历史与当前**。

---

## 核心设计

### 消息结构变更

当前结构（P003 后）：
```
[system] persona + summary
[user] # 历史上下文 ...（无"请输出 JSON"）      ← 历史1
[assistant] {"action": "silent"}
[user] # 历史上下文 ...                          ← 历史2
[assistant] {"action": "reply", ...}
... × 60轮 ...
[user] # 当前上下文 ... 请输出 JSON。             ← 当前请求
```

重构后结构：
```
[system] persona + summary
[user] # 历史对话记录（仅供参考，不要回复这些内容）
       [合并的历史消息纯文本，丢弃元信息]
[assistant] [最后一条历史 assistant 的精简回复，保持交替格式]
[user] # 近期对话
       [最后 N 轮原样保留，带元信息]
[assistant] [倒数第二条 assistant]
[user] # 当前上下文 ... 请输出 JSON。             ← 当前请求
```

### 三个关键变化

1. **历史合并为单个 user message**：从 60 个 user/assistant 对 → 1 个 user message（合并文本）+ 1 个 assistant placeholder
2. **历史中只保留语义内容**：丢弃群成员列表、时间戳元信息、状态行，只保留"谁说了什么"
3. **近期对话保留原格式**：最后 N 轮仍用"# 当前上下文"格式，保留完整元信息（当前请求需要）

### 合并后的历史文本格式

```
# 历史对话记录（仅供参考，不要回复这些内容）

[07-05 12:25] 忆雨(3052748978): 攻击性怎么这么强 没人唤醒你啊[@LubG]
[07-05 12:25] [bot]LubG: 我就说句安静，哪攻击了
[07-05 12:57] CuHe(2246894877): 神了
[07-05 12:59] [bot]LubG: silent
[07-05 13:15] 忆雨(3184546563): 今天好困
[07-05 13:16] [bot]LubG: 笑死，才下午就困
...
```

对比当前格式（每条历史带完整元信息）：
```
# 当前上下文
- 当前时间：2026-07-05 12:25 Sunday
- 你的昵称：LubG（QQ：3952966724）
- 状态：被动触发（有新群消息需要你看一眼）
- 群成员（昵称/QQ/角色/亲密度）：
忆雨​(3184546563) role=member affinity=3
...（7行成员列表）
# 最近群消息
[#1040221510] [12:25:36] 忆雨(3052748978): 攻击性怎么这么强...
请输出 JSON。
```

**收益估算**：
- 每条历史原 user content ~400-600 字 → 合并后每条 ~50-80 字（只保留发言）
- 60 轮历史从 ~30000 字 → ~4000 字，减少 80%+
- 注意力从分散在 120 条 messages → 集中在 1 条历史 + N 条近期

## 切分策略

### 推荐方案：固定轮数切分

**保留最后 N 轮作为"近期对话"，其余合并为"历史对话"**。

N 的选择（已确认）：
- **N=6 轮（12 条 messages）**：更保守，适合话题延续较多的场景。历史合并收益略减（70%+ token 减少），但近期上下文更完整。

切分点示例（假设 60 轮历史，N=6）：
```
messages[0:108]  → 合并为 1 个 user + 1 个 assistant（历史）
messages[108:120] → 原样保留（近期 6 轮）
messages[120]    → 当前请求
```

### 否决的方案

**K-means 聚类**：
- 过度工程化，话题边界不是时间窗口能刻画的
- K-means 需要预设 K 值，对话话题数不可预测
- 聚类结果可能不稳定，破坏前缀缓存

**LLM 判断话题边界**：
- 额外 API 调用增加延迟和成本
- 话题边界判断本身是模糊的，LLM 可能给出不一致结果
- 与前缀缓存的稳定性需求冲突

**多话题切分**：
- 复杂度暴增，收益不明显
- 多个 user message 块仍然会分散注意力（虽然比当前好）
- 摘要按话题分组是未来优化，当前不必要

### 切分理由

固定轮数切分的优势：
1. **零计算成本**：不需要聚类或 LLM 判断
2. **前缀缓存友好**：历史部分在两次压缩之间完全不变，缓存稳定
3. **可预测**：每次调用结构一致，便于调试
4. **足够用**：真人对话中 4 轮已覆盖绝大多数连贯话题，更早的内容作为"背景知识"足够

## 实现细节

### 改动位置

**`src/history.py`**：
- 新增 `get_messages_for_llm_merged()` 方法（或修改 `get_messages_for_llm()`）
- 不改 `conversation.json` 存储格式（仍按原样存储完整 messages）
- 不改 `append_turn`、`build_user_content` 等方法

### 合并逻辑

```python
def get_messages_for_llm_merged(self, recent_turns: int = 4) -> list[dict]:
    """返回合并后的 messages，历史部分合并为单条 user message。
    
    结构：
    [user] # 历史对话记录 ...（合并的文本）
    [assistant] [最后一条历史 assistant 的精简内容]
    [user/assistant 交替] × recent_turns（原样保留）
    [user] 当前请求（原样保留）
    """
    messages = self.messages.copy()
    if len(messages) <= recent_turns * 2:
        # 历史不足，无需合并
        return messages
    
    # 切分点：最后 recent_turns*2 条之前的是历史
    split_idx = len(messages) - recent_turns * 2
    history_msgs = messages[:split_idx]
    recent_msgs = messages[split_idx:]
    
    # 合并历史为单条文本
    history_text = self._merge_history_to_text(history_msgs)
    
    # 最后一条历史 assistant 的内容（保持 user/assistant 交替）
    last_history_assistant = None
    for i in range(len(history_msgs) - 1, -1, -1):
        if history_msgs[i]["role"] == "assistant":
            last_history_assistant = history_msgs[i]["content"]
            break
    
    # 组装结果
    result = [
        {"role": "user", "content": history_text},
    ]
    if last_history_assistant:
        # 精简 assistant 内容（只保留 thought 和 action，去掉冗余字段）
        result.append({"role": "assistant", "content": last_history_assistant})
    result.extend(recent_msgs)
    return result

def _merge_history_to_text(self, history_msgs: list[dict]) -> str:
    """把历史消息合并为可读文本，丢弃元信息只保留语义内容。
    
    格式：
    # 历史对话记录（仅供参考，不要回复这些内容）
    
    [07-05 12:25] 忆雨(3052748978): 攻击性怎么这么强
    [07-05 12:25] [bot]LubG: 我就说句安静
    ...
    """
    lines = ["# 历史对话记录（仅供参考，不要回复这些内容）", ""]
    
    for m in history_msgs:
        content = m["content"]
        if m["role"] == "user":
            # 从 user content 中提取群消息行
            # 格式：# 当前上下文\n...\n# 最近群消息\n<消息内容>\n请输出 JSON
            lines.extend(self._extract_messages_from_user_content(content))
        else:  # assistant
            # 从 assistant JSON 中提取 bot 的回复
            bot_text = self._extract_bot_reply_from_assistant(content)
            if bot_text:
                lines.append(bot_text)
    
    return "\n".join(lines)
```

### 元信息提取

`_extract_messages_from_user_content` 需要从原 user content 中提取：
- 时间（从"# 当前上下文"的"当前时间"行）
- 发言内容（从"# 最近群消息"节）

`_extract_bot_reply_from_assistant` 需要从 assistant JSON 中提取：
- 如果 action=reply/multi_reply：提取 messages 字段的文本
- 如果 action=silent：标记为 `[bot]LubG: silent` 或直接跳过
- 如果有 thought：可选保留（帮助 LLM 理解上下文）

### assistant placeholder 的处理

合并历史后，需要在历史 user message 后接一个 assistant message 保持交替格式。选择：
- **方案1**：用最后一条历史 assistant 的原始 JSON（推荐，信息完整）
- **方案2**：用精简版（只保留 thought + action + messages 文本）
- **方案3**：用占位文本 `{"action": "silent"}`（最简但损失信息）

推荐方案1：保持历史最后一条 assistant 的完整内容，让 LLM 知道"上一次我说了什么"，便于判断是否重复。

### 历史中 bot silent 回复的处理（已确认）

**跳过不显示**：历史文本中不出现 bot silent 的记录。
- 优点：历史更紧凑，只保留有意义的互动
- 缺点：LLM 不知道"我看过这条但没回"（可接受，真人也不会记得所有没回的消息）

实现：`_extract_bot_reply_from_assistant` 解析 assistant JSON 时，如果 action=silent，则不向历史文本追加任何内容。

### 前缀缓存影响

**需要仔细评估**。

重构后 messages 结构：
```
[system] 稳定
[user] 历史合并文本         ← 在两次压缩之间稳定
[assistant] 最后历史回复     ← 在两次压缩之间稳定
[user] 近期1                ← 每轮会变化（新的近期1变成历史）
[assistant] 近期1回复
...
[user] 当前请求             ← 每次变化
```

缓存命中范围：`system + 历史 user + 历史 assistant` 在两次压缩之间完全稳定，可命中前缀缓存。

与当前对比：
- 当前：`system + 60轮历史` 在两次压缩之间稳定，缓存命中范围大
- 重构后：`system + 1轮历史合并` 在两次压缩之间稳定，缓存命中范围小（但历史合并文本本身比 60 轮短很多，未命中部分 token 减少）

**净影响**：缓存命中范围缩小，但未命中的 token 总量也大幅减少（历史从 3 万字降到 4 千字）。总体 token 消耗应下降，缓存命中率可能略降但单次请求成本降低。

## 切分点随对话推进的变化

随着对话推进，切分点会移动：

```
第1轮：messages=[u1,a1]，无需合并
第2轮：messages=[u1,a1,u2,a2]，无需合并
...
第5轮：messages=[u1,a1,...,u5,a5]，split_idx=2，历史=[u1,a1]，近期=[u2,a2,u3,a3,u4,a4,u5,a5]
第6轮：messages=[u1,a1,...,u6,a6]，split_idx=4，历史=[u1,a1,u2,a2]，近期=[u3,a3,...,u6,a6]
```

**关键**：每次新增一轮，切分点前移，最旧的一条近期对话"毕业"进入历史。这条最旧的对话需要被合并到历史文本中。

这意味着 `get_messages_for_llm_merged()` 每次调用时都要重新合并历史，而不是缓存合并结果。但合并是纯字符串操作，成本可忽略。

## 与摘要压缩的协同

当前摘要压缩（方案A分层压缩）在 `history_limit / 2 = 120` 轮时触发，把中期 40 轮做 LLM 摘要、远期朴素摘要，保留近 30 轮原文。

重构后：
- **保留近 4 轮原文**（本方案的近期对话）
- **5-30 轮合并为历史文本**（本方案的历史合并）
- **30+ 轮仍走摘要压缩**（保持现有逻辑）

或者可以考虑调整压缩阈值：
- 近 4 轮：原样保留（近期对话）
- 5-30 轮：合并为历史文本（本方案）
- 30+ 轮：摘要压缩（现有逻辑）

这样三层结构：近期原文 → 历史合并 → 摘要。但增加了复杂度，建议初始版本不调整压缩逻辑，只在上层做合并。

2. 检查合并后的历史文本可读性（手动查看几条合并结果）

### 上线观察

1. 部署后观察"答非所问"是否完全消失
2. 关注 bot 是否能正确理解历史上下文（合并后历史信息密度降低，可能影响一些需要引用历史的场景）
3. 监控 token 消耗是否下降（预期下降 50%+）
4. 监控缓存命中率变化

### 回滚预案

`get_messages_for_llm_merged()` 是新方法，回滚只需让 `get_messages_for_llm()` 恢复原实现（或调用 P003 的标记逻辑）。

## 风险与权衡

### 风险

1. **历史信息密度降低**：合并后历史只保留"谁说了什么"，丢失了元信息（群成员列表、时间上下文）。可能影响一些需要历史元信息的判断
2. **近期对话边界感**：固定 4 轮切分可能不总是符合话题边界。如果话题跨越 5 轮，第 5 轮会被合并到历史，LLM 可能觉得"突然断了"
3. **合并逻辑的复杂性**：提取消息内容需要解析原 user content 格式，格式变动时需要同步更新提取逻辑

### 权衡

| 维度 | P003（标记+约束） | P004（合并重构） |
|---|---|---|
| 实施成本 | 低（改2个方法） | 中（新增合并逻辑） |
| 缓解效果 | 中（消除格式歧义） | 高（彻底分离历史与当前） |
| token 消耗 | 不变 | 下降 50%+ |
| 前缀缓存 | 基本不变 | 命中范围缩小但总量下降 |
| 可回滚性 | 极好 | 好 |
| 信息损失 | 无 | 丢失历史元信息 |

### 推荐实施顺序

1. **先做 P003**（方案 A+B）：快速缓解，验证"格式歧义"假设
2. **观察 1-2 天**：如果 P003 已经显著缓解问题，P004 可以缓做
3. **如果 P003 不够**：再做 P004，从结构上彻底解决

P003 和 P004 不冲突，可以叠加。P004 实施后，P003 的方案 A（标记历史）自然失效（因为历史已合并为单条 message），但方案 B（persona 约束）仍然有用。
