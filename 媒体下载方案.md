# 媒体下载方案：网络资源统一本地化（同步方案）

## 1. 背景与问题

当前 media 发送链路中，`data.url` 通道直接将 URL 传给 NapCat，由 NapCat 自行下载网络资源。导致：

1. **NapCat 10s 超时硬限**：NapCat 内部下载 URL 时有 10s 连接超时，修改需改源码重编译，成本极高。
2. **生成型 URL 必定超时**：AI 绘画（SDXL 30s+）、在线视频等资源，10s 远不够。
3. **重复浪费**：同一 URL 多次发送时 NapCat 每次重新下载（本方案暂不解决，留待未来缓存机制）。

## 2. 目标

将所有 `data.url` 通道统一改为：**Bot 主机同步下载到本地临时文件，再通过 file:// URI 发送给 NapCat**。

- NapCat 永远不接触网络资源，只读本地文件，毫秒级完成，彻底规避 10s 超时。
- manifest 结构不变（file / url 字段并存），sender 内部透明转换。
- 未来扩展 AI 生成内容只需暴露一个 GET 即返回内容的 URL，无需新增消息段类型。

## 3. 核心设计理念：同步 = 真人模拟

本方案**不引入异步下载**，采用完全同步的"下载 → 发送 → 记录"流程。理由：

**"像真人"的设计哲学**：真人在一个群里发图片时，需要先把图片准备好（下载/拍摄/选择），这期间不会跑去别的群说话。真人发完一个群才转到另一个群，**全局单队列的串行阻塞恰恰模拟了这种"专注一件事"的行为**。

异步下载虽然能避免 cycle 阻塞，但会带来更严重的问题：
- **消息顺序乱序**：multi_reply 中 text + image + text 组合，图片延后到达，群员看到的顺序与 LLM 记忆不一致，破坏"LLM 看到的发言顺序 = 真实群聊发言顺序"核心原则
- **历史状态不一致**：append 到 fast_buffer 时图片可能还没真正发出，下载失败会导致历史记录与实际发送不符
- **脱离上下文**：生成型 URL（30s+）下载完成时群聊已推进多轮，图片到达时已无语境

同步方案的代价是 cycle 阻塞（其他群触发排队入 `_cycle_queue`，cycle 结束后顺序处理），但：
- 不会丢消息，只是延后
- 普通图片/语音下载通常 1-3s，阻塞可接受
- 生成型 URL 阻塞较长，但符合"真人慢慢准备图片"的语义
- 代价远小于顺序乱序与状态不一致

## 4. 各 Sender 改造

### 4.1 统一改造模式

image / video / voice(url 通道) 三个 sender 的 `data.url` 分支统一改为：

```
data.url → 同步下载到本地临时文件 → file:// URI → NapCat（读本地文件，毫秒级）
data.file → 直接转 file:// URI（不变）
```

### 4.2 下载实现

新增 `src/media_downloader.py`，提供统一下载函数：

```python
def download_to_local(url: str, timeout: int = 30) -> Optional[str]:
    """下载 URL 到本地临时文件，返回本地路径。失败返回 None。

    Args:
        url: 网络资源 URL
        timeout: 下载超时（秒）
    Returns:
        本地文件路径，失败返回 None
    """
    ...
```

- **临时文件位置**：`media/downloaded/<sha256(url)[:16]>.<ext>`
- **扩展名推断**：优先从 `Content-Type`（如 `image/png` → `.png`），fallback 从 URL 路径后缀，再 fallback `.bin`
- **发送后删除**：sender 发送成功后删除临时文件（当前不做缓存，避免磁盘堆积；未来加缓存时改为保留即可）
- **失败处理**：下载失败返回 None，调用方据此跳过发送

### 4.3 sender 返回值改造

当前 `sender.send()` 返回 dict（NapCat 响应）。改造为返回 **bool**（True=发送成功，False=失败）：

```python
class NapCatImageSender:
    def send(self, group_id: str, data: dict) -> bool:
        url = data.get("url", "")
        file_path = data.get("file", "")
        temp_path = None
        if url:
            temp_path = download_to_local(url, timeout=30)
            if not temp_path:
                logger.warning(f"图片下载失败，跳过发送: url={url[:80]}")
                return False
            file_path = temp_path
        if file_path:
            file_uri = Path(file_path).resolve().as_uri()
            resp = self.client.send_group_msg(group_id, [{"type": "image", "data": {"file": file_uri}}])
            ok = bool(resp)  # _call 失败返回空 dict
            if temp_path and ok:
                _safe_delete(temp_path)  # 发送成功后删除临时文件
            elif temp_path and not ok:
                _safe_delete(temp_path)  # 发送失败也删除，避免残留
            return ok
        return False
```

video / voice(url) 同理。

### 4.4 _send_messages 改造

`main._send_messages` 检查 sender 返回值，**失败则不 append 到 fast_buffer**（保持历史事实性——历史只记录真实发生的事）：

```python
if seg.get("type") == "image":
    ok = self.image_sender.send(ctx.group_id, seg.get("data", {}))
    if not ok:
        logger.warning("图片发送失败，不记录到历史")
        continue  # 不 append，直接下一条
    handled = True
    break
```

## 5. 配置项

config.yaml 新增：

```yaml
media_download:
  timeout: 30          # 普通资源下载超时（秒）
  video_timeout: 60    # 视频资源下载超时（秒）
  temp_dir: "media/downloaded"  # 临时文件目录
```

## 7. 不做的事（明确排除）

- **缓存机制**：当前不实现。下载的临时文件发送后即删除。未来如需缓存，改为保留文件 + 维护 url→path 映射即可。
- **异步下载**：不实现。同步阻塞符合"真人模拟"设计哲学，避免顺序乱序与状态不一致。
- **fallback 到原 url**：不实现。NapCat 10s 必定超时，fallback 无意义。

## 8. 关于第 5 点扩展性的说明

未来 AI 绘图/视频可通过写客户端+服务端暴露 API，通过 URL 统一接口请求。本方案完全兼容：

- 生成型 URL 只需满足"GET 即返回媒体字节"的契约
- 调用方在 sender 中传入更长 timeout（如 draw 场景传 timeout=600）
- 无需新增消息段类型，LLM 仍输出 `{"type": "image", "data": {"url": "..."}}`
- 唯一注意：生成型 URL 同步下载会阻塞 cycle 较长（如 draw 30s+），但符合"真人慢慢生成图片"的语义，可接受
