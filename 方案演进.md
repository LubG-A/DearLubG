# 方案演进

本文档记录 DearLubG 机器人从单群架构到多群运行的演进过程。每个阶段独立设计、独立验证，前序阶段是后序阶段的基础。

---

## 演进总览

| 阶段 | 名称 | 状态 | 核心目标 | 关键产出 |
|---|---|---|---|---|
| A | GroupContext 容器抽取 | ✅ 已落地 | per-group 状态隔离，为多群做结构准备 | `src/group_context.py` |
| B | 多群配置与全局串行队列 | ✅ 已落地 | 真正的多群运行 + cycle 队列 | config `group_ids` 列表 + `_cycle_queue` |
| C | 跨群印记板 | ❌ 已撤销 | 跨群感知（"别的群在聊什么"） | 见下方"阶段 C 撤销说明" |
| D | 主动触发多群化 | ✅ 已落地 | per-group 独立主动触发倒计时 | ActiveScheduler 移入 GroupContext |

**演进前**（阶段 A 之前）：单群运行，所有状态在 `Bot` 类的 `self` 上，config 只有一个 `group_id`，状态文件平铺在 `state/` 根目录。

**演进后**（阶段 D 完成后）：多群独立运行，per-group 状态封装在 `GroupContext` 中，全局共享单例（napcat/llm/affinity/persona/senders）留在 `Bot`，cycle 串行 + 全局队列保证多群有序处理，per-group 主动触发调度器独立运行。

## 阶段 A：GroupContext 容器抽取

**文档**：`阶段A_GroupContext重构.md`

### 背景动机

单群架构下，`Bot` 类承载所有状态（conversation、attribution、trigger_evaluator、last_reply_time、quiet_timer、cycle_pending），无法支持多群。需要把 per-group 状态抽取到独立容器，为多群做结构准备。

### 核心设计决策

- **A1**：GroupContext 作为**数据容器**（非带方法的完整对象）——最小改动搭结构骨架，方法留 `Bot` 接受 `ctx` 参数。避免一次性大重构的风险。
- **A2**：HistoryManager/AttributionManager 文件名**暂不参数化**——阶段 A 保持 `conversation.json`/`state.json` 不变，阶段 B 再加子目录。
- **A3**：NapCatClient**暂不重构多群**——阶段 A 保持单群，GroupContext 不持有 NapCatClient（全局共享）。
- **A4**：`_cycle_pending` 移入 GroupContext（per-group），`_cycle_running` 保留 Bot（全局串行）。
- **A5**：状态归属表——conversation/attribution/trigger/last_reply_time/quiet_timer/cycle_pending 移入 GroupContext；affinity/napcat/llm/persona/senders/scheduler/cycle_running 留 Bot。
- **A6**：GroupContext **持有共享单例引用**（napcat/llm/affinity），非通过 Bot 中转。

### 改动范围

- **新增**：`src/group_context.py`（GroupContext 数据容器类）
- **修改**：`main.py`（10+ 方法加 ctx 参数，`self.groups: dict[str, GroupContext]`）
- **不改**：history/attribution/affinity/trigger/napcat_client/persona/scheduler 等核心模块

### 验证标准

纯重构，零行为变化。单群运行，所有可观测行为与重构前一致。

## 阶段 B：多群配置与全局串行队列

**文档**：`阶段B_多群配置与全局串行队列.md`

### 背景动机

阶段 A 搭好了 per-group 容器骨架，但实际仍是单群运行（config 只有一个 group_id，状态文件平铺）。需要实现真正的多群支持：多群配置、多群收发、多群文件隔离，以及多群 cycle 串行队列。

### 核心设计决策

- **B1**：config.yaml 群配置采用**纯列表 `group_ids: [list]`**，废弃 `group_id`，无兼容层。
- **B2**：per-group 文件存储采用**子目录方案**（`state/{group_id}/conversation.json`），非文件名后缀。手动迁移，不写迁移代码。
- **B3**：NapCatClient 多群化——`__init__` 接受 `group_ids: list[str]`，`member_cache` 嵌套化 `{group_id: {qq: info}}`，所有 API 方法接受 `group_id` 参数。单实例 + group_id 参数（非多实例）。
- **B4**：webhook 去掉 `target_group_id` 过滤，按 `data.group_id` 路由到对应 ctx，不在 groups 字典则丢弃。
- **B5**：**全局 cycle 队列**——`_cycle_queue: deque[str]` + `_cycle_queue_set: set[str]`（去重）。撞 cycle 时入队，cycle 结束后 `_drain_cycle_queue` 迭代处理（非递归）。重跑不传 soft_factors。
- **B6**：阶段 B**关闭主动触发**（`active_trigger.enabled: false`），代码保留但 `on_active_trigger` 仅改引用为 `group_ids[0]`，不实现轮询/选群。
- **B7**：NapCatClient 方法签名**直接改，不做兼容层**——一次性改完所有调用点。

### 改动范围

- **修改**：`src/config.py`（NapCatConfig 用 group_ids 替换 group_id）
- **修改**：`src/napcat_client.py`（多群化：__init__/warmup/get_nickname/所有 API 方法加 group_id）
- **修改**：`src/group_context.py`（__init__ 传入 per-group state_dir）
- **修改**：`main.py`（cycle 队列、warmup 循环创建 GroupContext、webhook 路由、_drain_cycle_queue）
- **不改**：history/attribution/affinity/trigger/persona/scheduler/llm_client 等核心模块

### 验证标准

1. 不同群被动触发正常工作
2. 大消息量下不同群维护正常回复顺序
3. 长期任务执行下维护独立的 conversation 和摘要信息

### 状态目录结构变化

```
├── conversation.json          ├── {group_id}/
├── state.json                 │   ├── conversation.json
```

## 阶段 C：跨群印记板（已撤销）

### 历史背景

阶段 C 曾计划实现跨群感知：让 bot 在 A 群聊过的话题，到 B 群能"有印象"。方案为紧凑印记板（`CrossGroupImpressionStore`），存储每群的群印象+人印象，惰性更新（每群 trigger_count >= 5 触发 LLM 摘要），注入到 system prompt 的"# 其他群的近况"节。

### 撤销原因：DeepSeek 前缀缓存匹配效率暴跌

**部署测试发现**：阶段 C+D 上线后，API 请求的**未命中输入缓存比例从约 3.4% 飙升至 46.3%**，消费金额大幅提高。

**根因分析**：

DeepSeek 的前缀缓存机制按 messages 数组从开头逐 token 匹配，**第一个变化点之后的所有内容全部失效**。

原先利于缓存的消息结构：
```
[人格+协议+背景](system, 稳定) + [早期摘要](system, 低频变) + [近期60轮历史原文](messages, 稳定) + [当前对话](messages, 不可避免变化)
```
其中占据极大篇幅的**近期60轮历史原文完全稳定**，是前缀缓存的最大受益者，命中率极高。

阶段 C 插入 impression 后的结构：
```
[人格+协议+背景] + impression(每次变) + [早期摘要] + [近期60轮历史原文] + [当前对话]
```

`others_impressions` 几乎每次调用都在变化（不仅每5次触发更新内容，相对时间戳如"2分钟前"→"3分钟前"每次都重新计算）。impression 一变，从变化点开始 → 早期摘要 + **所有60轮历史原文** + 当前消息全部失效。

**关键认知**：任何"会变化的内容"放在 system prompt 里，都会摧毁其后所有历史消息的缓存。把 impression 调整到 summary 之后只能保住 summary（最多2000字），保不住60轮历史原文（篇幅远大于 summary），杯水车薪。

**结论**：要想保住历史原文缓存，system prompt 必须完全稳定。阶段 C 的设计与前缀缓存机制根本冲突，撤销是唯一正确选择。

### 撤销操作

通过 `git revert 3613fdd`（阶段C实施提交）完成。撤销内容：
- 删除 `src/impression.py`
- `main.py`：移除 `impression_board` 初始化、import、warmup 日志、`_run_llm_cycle` 中的 `others_impressions` 调用与 `maybe_update` 调用
- `src/persona.py`：`render_system_prompt` 恢复原签名（无 `others_impressions` 参数）、移除规则17
- `src/llm_client.py`：移除 `summarize_for_impression` 方法
- `src/history.py`：移除 `get_recent_user_contents` 方法

阶段 D 的 scheduler 改动**完整保留**，因为阶段 C 和阶段 D 改的是完全不同的代码区域，互不依赖。

### 跨群感知的未来方向

跨群感知的需求仍然存在，但实现方式需要避开前缀缓存冲突。可能的未来方向：
- 注入到 user content 而非 system prompt（user content 本来每次就变，不影响缓存）
- 作为独立的 LLM 调用（后台异步生成印记，不影响主对话的缓存）
- 这些方向均未实施，留待未来评估

## 阶段 D：主动触发多群化

**文档**：`阶段D_主动触发多群化.md`

### 背景动机

阶段 B 为了聚焦多群配置和 cycle 队列，关闭了主动触发（`enabled: false`），`on_active_trigger` 硬编码 `group_ids[0]`。现在需要补齐主动触发多群化：每群独立倒计时。

### 核心设计决策

- **D1**：调度器归属为**per-group 实例，移入 GroupContext**（方案 B），非全局单例 + per-group 状态字典。与 `_quiet_timer` 模式一致。
- **D2**：间隔计算**沿用原有随机 min-max**（`_generate_next_check_time` 不改），活跃度自适应留待未来。
- **D3**：主动触发到期时**直接 is_active=True**，不判断 pending 状态——cycle 串行机制兜底撞锁入队。
- **D4**：状态持久化为**per-group `state/{group_id}/scheduler.json`**，非全局文件。不迁移旧数据（阶段 B 期间已关闭，旧数据过时）。
- **D5**：`on_active_trigger` 改为**接受 group_id 参数**，由 per-group 调度器 Timer 直接传入。回调签名从 `Callable[[], None]` 改为 `Callable[[str], None]`。
- **D6**：**保留深夜跳过逻辑**（23:00-3:00 不触发，顺延到次日 3:00）。
- **D7**：启动初始化——**warmup 时为每群启动调度器**（遍历 self.groups），enabled=false 时不启动。

### 改动范围

- **修改**：`src/scheduler.py`（__init__ 新增 group_id 参数、start 接受 group_id、_fire 传 group_id 给回调、日志加 group_id 前缀）
- **修改**：`src/group_context.py`（__init__ 新增 `self.active_scheduler`）
- **修改**：`main.py`（删除全局 scheduler 单例、warmup 中为每群启动调度器、on_active_trigger 改为接受 group_id、run() 删除全局 scheduler 启动）
- **修改**：`config.yaml`（`active_trigger.enabled: true` 开启）

### 验证标准

1. 每群独立倒计时，`state/{group_id}/scheduler.json` 各自独立
2. 活跃群和冷清群用同一随机区间，各自独立到期
3. 深夜不触发
4. cycle 撞锁时正确入队

### 状态目录结构变化

```
演进前（阶段 B）:              演进后（阶段 D）:
state/                         state/
├── {group_id}/                ├── {group_id}/
│   ├── conversation.json      │   ├── conversation.json
│   ├── state.json             │   ├── state.json
│   └── attribution_log.jsonl  │   ├── attribution_log.jsonl
├── affinity.json              │   └── scheduler.json      # per-group（新增）
```

## 当前架构总览（阶段 D 完成后，阶段 C 已撤销）

### Bot 全局共享单例

| 组件 | 说明 |
|---|---|
| `napcat` | NapCatClient，多群 HTTP 客户端（member_cache 嵌套化） |
| `llm` | LLMClient，大模型 API 调用（chat + summarize） |
| `affinity` | AffinityManager，亲密度管理（全局跨群，按 QQ） |
| `persona_renderer` | PersonaRenderer，系统提示词渲染 |
| `message_sender` / `ai_voice_sender` / `emoji_reactor` | 发送器（全局共享） |
| `_cycle_running` / `_cycle_queue` | LLM cycle 串行控制（全局） |

### GroupContext per-group 状态

| 组件 | 说明 |
|---|---|
| `history` | HistoryManager，多轮对话历史（messages + summary + pending + delayed_replies） |
| `attribution` | AttributionManager，归因自校准（per-group dynamic_weight） |
| `trigger_evaluator` | TriggerEvaluator，评分门槛（使用本群 history + 全局 affinity） |
| `active_scheduler` | ActiveScheduler，主动触发调度器（per-group 独立倒计时） |
| `last_reply_time` | 本群冷却计时 |
| `_quiet_timer` | 本群静默窗口定时器（150s） |
| `_cycle_pending` | 本群撞 cycle 重试标志 |

### 状态文件布局

```
state/
├── {group_id}/                    # per-group 子目录
│   ├── conversation.json          # 多轮对话历史
│   ├── state.json                 # dynamic_weight + round_count
│   ├── attribution_log.jsonl      # 归因日志
│   └── scheduler.json             # 主动触发定时器状态（per-group）
├── affinity.json                  # 亲密度（全局跨群）
├── background_story.md            # 背景故事原文
└── background_summary.json        # 背景故事总结缓存
```

### 消息构造与前缀缓存

当前消息结构充分利用 DeepSeek 前缀缓存机制：

```
messages = [
    {"role": "system", "content": [人格+协议+背景](稳定) + [早期摘要](仅压缩时变)},
    *history_messages,          # 近60轮历史原文（稳定，最大篇幅，缓存主要受益者）
    {"role": "user", ...}       # 当前对话（每次变化，不可避免）
]
```

- **system prompt**：template 完全稳定，summary 仅在分层压缩时更新（低频），构成稳定前缀
- **history_messages**：近60轮历史原文在两次压缩之间完全稳定，是前缀缓存的最大受益者
- **new_user_content**：每次新增，不可避免要重新计算

**设计原则**：system prompt 中不放置任何高频变化的内容（这是阶段 C 撤销的核心教训）。

### 并发模型

- **接收线程**（webhook）：只写 `ctx.fast_buffer`（`_buffer_lock` 保护，持锁极短），不持任何锁
- **LLM 工作线程**：`_cycle_running` 全局标志保证串行，LLM 调用和发送不持锁
- **撞 cycle 处理**：同群注册 `ctx._cycle_pending`（cycle 内重跑），跨群入 `_cycle_queue`（cycle 后 `_drain_cycle_queue` 处理）
- **per-group 定时器**：`_quiet_timer`（150s 被动兜底）和 `active_scheduler`（随机 min-max 主动）各自独立，靠 cycle 串行机制去重

## 未来方案（已记录，当前不实现）

- **跨群感知（替代方案）**：原阶段 C 的印记板方案因前缀缓存冲突已撤销。未来如需实现跨群感知，应避开 system prompt 注入（改注入到 user content，或作为独立 LLM 调用），避免破坏前缀缓存。
- **群活跃度自适应**：根据群近期消息频率动态调整主动触发间隔，活跃的群多触发，冷清的群少触发
- **亲密度驱动的主动触发间隔**：群内亲密度高的人多时缩短间隔
- **per-group 深夜时段配置**：不同群配置不同的静默时间
- **主动触发时注入群活跃度到 prompt**：让 LLM 知道"这个群最近很活跃/冷清"（注意：需注入到 user content 而非 system prompt，避免前缀缓存冲突）
- **联网新闻获取**：主动触发时拉取热点新闻作为话题素材
- **长时间延迟回复的引用**：延迟回复跨越较长时间时自动添加引用段
- **主动触发的内容素材库**：维护 bot 的想法/槽点库，避免每次主动开口都是"突然想起来"
