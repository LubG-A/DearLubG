# 方案演进与未采纳方案

本文档记录方案演进过程与未采纳的备选方案，便于后续回溯与决策复盘。
**当前最终实现以 `任务描述.md` 为准**，本文档仅作历史记录。

---

## 一、版本演进

### V0 原始方案（最初版本）
- 历史记录格式：`[USER123456789]：你好` / `[MODEL123456789]：你好...`
- 触发机制：被 @ 或累计消息阈值（二选一）
- 输出协议：`[TO-USERX]` / `[NOTHING-TO-SAY]` 文本标签
- 回复类型：纯文本

### V1 第一轮迭代
- 历史记录：带时间戳+昵称的群聊视图
- 触发机制：评分门槛（替代原 @/阈值）
- 输出协议：JSON，`action=silent/reply/multi_reply/react`
- 回复类型：text/at/reply/face/image/voice/forward
- 新增 Persona 卡与系统提示词模板

### V2 最终敲定（当前实现）
- 历史记录：**多轮对话格式**（标准 user/assistant 交替，非"一坨文本"）
- 触发机制：**两层架构**（评分门槛 + LLM 自决 silent），评分参数由**归因机制**自校准
- 输出协议：同 V1，新增 `affinity_delta` 字段
- voice：**双通道**（`/send_group_ai_record` + 本地音频转 OB11MessageRecord），不耦合 TTS 引擎
- image：**空实现**，预留接口
- 亲密度：LLM 输出 `affinity_delta` 微调
- **冷启动保护**：前 `cold_start_rounds` 轮不更新 `dynamic_weight`，仅记日志
- **模块接口设计**：显式定义 `MessageSender`/`VoiceSender`/`ImageSender`/`EmojiReactor` 四个协议接口
- **亲密度初值全 0**：不预设管理员/普通成员/陌生人的差异化初值，避免后续回溯设计依据
- **迭代顺序**：text-only → 归因+亲密度 → voice → image+emoji，分四阶段

## 二、未采纳的备选方案

### 1. 用纯评分机制替代 silent 判断
**方案描述**：完全用评分阈值决定是否回复，不调 LLM silent。

**不选原因**：评分只能做关键词匹配，无法理解语义。原方案 `[NOTHING-TO-SAY]` 的精髓是把语义判断权交给 LLM，让模型像真人一样"看完再说有没有话说"。

**最终采用**：两层架构，评分只决定"要不要让 LLM 看一眼"，回复与否由 LLM 自决。

### 2. 直接敲定 TTS 引擎（如 edge-tts）
**方案描述**：脚本直接集成 edge-tts 或 napcat 自带 TTS。

**不选原因**：违反模块解耦原则。voice 应作为可插拔能力，由 `VoiceSender` 策略模式隔离。

**最终采用**：voice 走两条通道——NapCat 自带的 `/send_group_ai_record`，和本地音频文件转 OB11MessageRecord。后续如需 TTS 引擎，作为新的 `VoiceSender` 实现插入，不修改主流程。

### 3. 在线状态切换模拟真人作息
**方案描述**：调用 `/set_online_status` 偶尔切换"离开/忙碌"，制造"真人不一定在看"的感觉。

**不选原因**：服务器晚上会关，无需模拟"离线"。

**最终采用**：不实现。

### 4. emoji 反应立即启用
**方案描述**：JSON 协议里直接支持 `action=react` 并落地。

**不选原因**：emoji_id 白名单与 image 系统共用"消息资源管理"逻辑，需统一处理。

**最终采用**：先完成 image 系统（包括读取和发送），再处理 emoji 部分。JSON 协议字段保留，脚本侧 `EmojiReactor` 接口预留。

### 5. Persona 卡内容在开发阶段敲定
**方案描述**：在方案文件里写死"林夏"的具体人格设定。

**不选原因**：Persona 内容是部署阶段的配置问题，与开发阶段解耦。

**最终采用**：开发阶段只保证框架支持可配置的 Persona 注入，实际内容由 `config.yaml` 决定，不在方案文档里敲定。

### 6. 亲密度冷启动差异化初值
**方案描述**：群管理员初值 60、普通成员 30、陌生人 10，让 LLM 一开始就有"熟人/陌生人"的差异化依据。

**不选原因**：不属于当前开发阶段的问题。预设具体数值后续可能反过来回溯"当时为什么设这些值"，缺乏设计依据。

**最终采用**：亲密度初值统一为 0，由 LLM 通过 `affinity_delta` 在运行中逐步建立差异化。

### 7. 历史记录"一坨文本"格式
**方案描述**：每次 LLM 调用都把全部历史拼成一个 user content，无 assistant role。bot 回复只是文本塞回 buffer。

**不选原因**：
1. LLM 无法区分"我说的"和"别人说的"，对话连贯性差。
2. 未利用 LLM 的多轮对话理解能力（标准 user/assistant 交替格式下表现更优）。
3. 历史压缩只能按条数截断，无法按轮次压缩。

**最终采用**：标准多轮对话格式。每轮"看一眼"触发时拼一个 user，LLM 返回的完整 JSON（含 thought，无论 silent 还是 reply）作为 assistant 落地。silent 也是真实回复，不存在"连续两个 user"的问题。

### 8. silent 时不落地 assistant（连续 user 方案）
**方案描述**：silent 时不插入 assistant，把后续群消息合并进上一个 user。

**不选原因**：LLM 只要被调用必有回复，silent 是我们定义的状态而非 LLM 的能力。silent 的回复也含思考过程（thought），对 LLM 自己后续推理有连贯价值（"我上轮觉得张三在吹牛"）。不落地 assistant 会丢失这一信息。

**最终采用**：silent 也正常落地 assistant，content 是完整 JSON（含 thought）。

### 9. 定时提醒/定时任务能力
**方案描述**：支持 LLM 创建定时任务（如"5分钟后提醒我喝水"），到点主动发起消息。

**不选原因**：
1. **与设计目标冲突**：bot 唯一目标是"像真人一样交互"。真人面对"5分钟后提醒我喝水"会答应但可能忘，不会像闹钟一样精准定时。定时任务会让 bot 显得像工具而非群友。
2. **架构能力缺失**：`reply_delay_minutes` 是"延迟回复当前消息"（bot 作为回复方），不是"定时发起任务"（bot 作为主动方）。当前架构没有"LLM 创建定时任务"的输出协议字段，也没有独立的定时任务调度器。
3. **人设偏移**：定时任务会让 bot 从"群友"偏移为"助理"，偏离核心设计哲学。

**最终采用**：不实现定时任务能力。bot 在这类请求上表现得"像真人一样不靠谱"（答应但可能忘）是可接受的，甚至是设计意图的一部分。

### 10. multi_reply 整块 append 到 fast_buffer
**方案描述**：把整个 multi_reply 当作原子块 append 到 fast_buffer，避免中间夹入群成员消息。

**不选原因**：真人边打字别人边插话是真实场景。整块 append 会丢失这种真实感，让 bot 的连发显得"原子化"而非"边想边打字"。

**最终采用**：multi_reply 逐条 append，间隔期间（0.8-2.5s 随机）群成员插话自然夹在 bot 消息之间（见决策 16）。

### 11. 用评分扣分替代冷却机制
**方案描述**：用"自身最近 60 秒已回过 -15"评分因子抑制密集触发，而非冷却直接跳过。

**不选原因**：评分扣分仍需调用 LLM 判断（只是分数变低），无法真正抑制密集触发。冷却机制直接跳过触发更直接有效，且实现更简单。

**最终采用**：三路触发决策 + 冷却机制（见决策 17）。原方案表的 -15 因子保留在 config.yaml 但实际未使用，由 `SOFT_COOLDOWN_MIN` 替代。

## 三、设计决策记录

### 决策 1：评分参数采用归因自校准
- 触发参数不再写死，而是根据 LLM 返回结果反馈调整。
- 每个软因子有 `base_value`（如兴趣 +25）与 `dynamic_weight`（初始 0，baseline）。
- 有效贡献 = `base_value + dynamic_weight`。
- LLM silent 时：`dynamic_weight -= v * step`（`v` 是本轮该因子的实际贡献）。
- LLM reply 时：`dynamic_weight += v * step * 0.5`（温和上调）。
- `step` 默认 0.05，幅度小，避免震荡。
- 限制：高分硬触发（@、提问）**不参与**归因；软因子（兴趣/熟人/话题热）参与。`dynamic_weight` 上下界 clip（默认 `[-base_value, +base_value]`），避免无界漂移。

### 决策 2：voice 双通道
- **通道 A**：`/send_group_ai_record`（NapCat 内置 AI 角色，零依赖）。
- **通道 B**：本地音频文件 → OB11MessageRecord 段 → `/send_group_msg`。
- 通过 `VoiceSender` 接口隔离，未来可插入 edge-tts / 第三方 TTS。

### 决策 3：image 空实现
- 协议保留 image 段，脚本侧 `ImageSender` 抛 `NotImplementedError` 或仅记日志。
- 后续根据实际需求（拉 URL / 描述触发由脚本拉图）再实现具体策略。

### 决策 4：亲密度由 LLM 微调
- LLM 输出可选字段 `affinity_delta: {qq: delta}`，`delta ∈ [-2, +2]`。
- 每次很小变化，模拟"生活中没有一蹴而就的事情"。
- 脚本侧维护持久化 `{qq: affinity}`，超界 clip 到 `[0, 100]`。

### 决策 5：Persona 内容不在开发阶段敲定
- Persona 卡是配置项，开发阶段只保证框架支持。
- 实际内容（姓名/性格/口癖）由部署阶段决定。

### 决策 6：归因机制加冷启动保护
- 前 `cold_start_rounds` 轮（默认 20）不更新 `dynamic_weight`，仅写入 `attribution_log.jsonl`。
- 避免初始随机行为污染参数。

### 决策 7：显式定义 4 个协议接口
- `MessageSender` / `VoiceSender` / `ImageSender` / `EmojiReactor`。
- 主流程只依赖接口不依赖实现，后续替换实现时主流程零修改。
- 当前阶段：`MessageSender` 真实实现，`VoiceSender` 双通道实现，`ImageSender` 空实现，`EmojiReactor` 预留实现。

### 决策 8：亲密度初值统一为 0
- 不预设管理员/普通成员/陌生人的差异化初值。
- 由 LLM 通过 `affinity_delta` 在运行中逐步建立差异化，避免后续回溯设计依据。

### 决策 9：voice 失败回退作为备选项
- QQ AI 语音调用一般很稳定，失败回退默认关闭。
- 如需启用，`config.yaml` 配置 `voice.fallback_to_text: true`，失败时降级为 text 段。

### 决策 10：分四阶段迭代开发
- 阶段一：text-only 朴素版本（跑通主流程 + 接口骨架）。
- 阶段二：归因机制与亲密度持久化。
- 阶段三：voice 双通道。
- 阶段四：image 系统与 emoji 反应。
- 每阶段完成后做回归测试。

### 决策 11：历史记录采用标准多轮对话格式
- messages 严格 user/assistant 交替，非"一坨文本"。
- 每轮"看一眼"触发时拼一个 user，pending 群消息批次进入。
- LLM 返回的完整 JSON（含 thought，无论 silent 还是 reply）作为 assistant 落地。
- silent 也是真实回复，不存在"连续两个 user"的问题。
- 早期对话压缩成摘要拼到 system 末尾，让 LLM 仍能记得"之前看过什么"。
- 触发条件：user/assistant 对数 ≥ `history_limit / 2`（默认 20 轮）。
- 保留近 `history_keep_recent / 2` 轮（默认 10 轮）原文。

### 决策 12：assistant 落地完整 JSON（含 thought）
- thought 虽不发送到群里，但对 LLM 自己后续推理有连贯价值。
- token 成本可控，且能保留 LLM 的"内心 OS"。
- 不做精简，直接落地 LLM 返回的原始字符串。

### 决策 13：bot 回复进入 pending，不直接进入 messages
- bot 实际发送的文本写入 `pending_group_msgs`（前缀 `[bot]`）。
- 待下一轮"看一眼"触发时拼入下一个 user，形成闭环。
- 这样 LLM 能看到"我上轮说了什么"，与真人记忆一致。

### 决策 14：bot 回复改为进入 fast_buffer（修订决策 13）
- **背景**：决策 13 让 bot 回复进 pending，但 pending 在 LLM 调用前已 drain，bot 回复写入时 pending 已被 consume 清空，导致 bot 回复被排到下一轮 pending 的队首，而 LLM 调用期间到达的群消息排在后面——时间戳顺序与排列顺序矛盾。
- **修订**：bot 回复改为写入 `fast_buffer`（`is_bot=True` 标记），与群消息按真实 append 顺序混合。下一轮 drain 时一起进入 pending，保证发言顺序=真实群聊发言顺序。
- **影响**：`append_group_message` 新增 `is_bot` 参数，成为接收线程与 LLM 线程的统一入口；删除原 `append_bot_reply_to_pending`；`recent_message_count` 过滤 `is_bot=True` 避免 bot 自我催化话题热度。
- **并发**：fast_buffer 现有两个生产者（接收线程 + LLM 线程），都通过 `_buffer_lock` 保护；锁获取顺序一致（`_llm_lock` 外层 → `_buffer_lock` 内层），无死锁。

### 决策 15：软因子触发接入与 hit 标记修复
- **背景**：原代码 `should_peek()` 从未被调用，所有非硬触发走静默窗口兜底，软因子评分不参与触发决策，归因系统完全休眠。
- **修复**：`on_group_message` 新增 `elif should_peek(score)` 分支，软因子过阈值时立即触发，携带 `soft_factors` 供归因使用。
- **hit 标记**：原代码 `interest_keyword`/`topic_hot` 创建时 `hit` 恒为 False，`soft_factors` 过滤条件 `f.effective_value != 0` 导致未命中因子（base_value≠0）也进入归因被错误调权。修复为正确传 `hit` 参数，过滤条件改为 `f.hit`——只有命中的软因子才参与归因。
- **peek_threshold 调参**：根据日志评分分布选定 `peek_threshold=10`：interest_keyword 命中即触发（深夜+15≥10），非深夜 topic_hot 命中也触发（+10≥10），深夜单独 topic_hot 不触发（0<10，靠静默窗口兜底）。

### 决策 16：multi_reply 逐条 append 到 fast_buffer
- **方案**：multi_reply 的多条 bot 消息逐条 append 到 fast_buffer，不作为原子块。
- **原因**：真人边打字别人边插话是真实场景，逐条 append 让 multi_reply 间隔期间（0.8-2.5s 随机）群成员插话自然夹在 bot 消息之间，更符合真实发言顺序。
- **替代方案**：整块 append（把整个 multi_reply 当作原子块）——未采纳，因为会丢失"边打字边有人插话"的真实感。

### 决策 17：三路触发决策与冷却机制
- **背景**：原方案"评分门槛决定看一眼"语义模糊，未明确低分消息如何处理、密集触发如何抑制。
- **采用**：三路触发决策——硬因子立即（5s 冷却）/ 软因子立即（过 peek_threshold，16-60s 冷却）/ 静默窗口兜底（20s 无新消息）。
- **冷却替代评分扣分**：原方案表的"自身最近 60 秒已回过 -15"未作为评分因子实现，改由 `SOFT_COOLDOWN_MIN` 冷却机制替代——冷却直接跳过触发，比扣分更直接。
- **归因范围**：只有软因子立即路径传 `soft_factors`，归因生效；硬因子和静默窗口兜底路径传 None，归因跳过（无明确触发因子可归因）。

### 决策 18：字数限制与 multi_reply 长度放宽
- **字数限制**：原方案"其余 1-8 字"调整为"其余 1-11 字"。8 字太严，11 字更符合中文短回应习惯。
- **multi_reply 长度**：原方案"2-4 条"放宽到"1-10 条"。真人连发短消息场景可能超过 4 条（如连续吐槽），reply 和 multi_reply 统一为 1-10 条，仅按 action 区分语义。

### 决策 19：亲密度数据结构增加时间戳
- **原方案**：`{qq: affinity}`（纯数值）。
- **修订**：`{qq: {"value": int, "last_updated": ISO时间戳}}`。
- **原因**：时间戳便于调试和审计亲密度变化历史，不增加逻辑复杂度。
