# 方案演进

本文档记录 DearLubG 机器人从单群架构到多群运行的演进过程。每个阶段独立设计、独立验证，前序阶段是后序阶段的基础。

---

## 演进总览

| 阶段 | 名称 | 状态 | 核心目标 | 关键产出 |
|---|---|---|---|---|
| A | GroupContext 容器抽取 | ✅ 已落地 | per-group 状态隔离，为多群做结构准备 | `src/group_context.py` |
| B | 多群配置与全局串行队列 | ✅ 已落地 | 真正的多群运行 + cycle 队列 | config `group_ids` 列表 + `_cycle_queue` |
| C | 跨群印记板 | ❌ 已撤销 | 跨群感知（"别的群在聊什么"） | 见下方"阶段 C 撤销说明" |
| D | 主动触发多群化 | ✅ 已落地 | per-group 独立主动触发倒计时 | ActiveScheduler 移入 GroupContext |

**演进前**（阶段 A 之前）：单群运行，所有状态在 `Bot` 类的 `self` 上，config 只有一个 `group_id`，状态文件平铺在 `state/` 根目录。

**演进后**（阶段 E 完成后）：多群独立运行，per-group 状态封装在 `GroupContext` 中，全局共享单例（napcat/llm/affinity/persona/senders）留在 `Bot`，cycle 串行 + 全局队列保证多群有序处理，per-group 主动触发调度器独立运行。多媒体发送能力完整（voice 三通道 + image/video 双通道 + 网络资源统一下载 + AI 绘画接口），素材库作为稳定能力注入 system prompt。

## 阶段 A：GroupContext 容器抽取

**文档**：`阶段A_GroupContext重构.md`

### 背景动机

单群架构下，`Bot` 类承载所有状态（conversation、attribution、trigger_evaluator、last_reply_time、quiet_timer、cycle_pending），无法支持多群。需要把 per-group 状态抽取到独立容器，为多群做结构准备。

### 核心设计决策

- **A1**：GroupContext 作为**数据容器**（非带方法的完整对象）——最小改动搭结构骨架，方法留 `Bot` 接受 `ctx` 参数。避免一次性大重构的风险。
- **A2**：HistoryManager/AttributionManager 文件名**暂不参数化**——阶段 A 保持 `conversation.json`/`state.json` 不变，阶段 B 再加子目录。
- **A3**：NapCatClient**暂不重构多群**——阶段 A 保持单群，GroupContext 不持有 NapCatClient（全局共享）。
- **A4**：`_cycle_pending` 移入 GroupContext（per-group），`_cycle_running` 保留 Bot（全局串行）。
- **A5**：状态归属表——conversation/attribution/trigger/last_reply_time/quiet_timer/cycle_pending 移入 GroupContext；affinity/napcat/llm/persona/senders/scheduler/cycle_running 留 Bot。
- **A6**：GroupContext **持有共享单例引用**（napcat/llm/affinity），非通过 Bot 中转。

### 改动范围

- **新增**：`src/group_context.py`（GroupContext 数据容器类）
- **修改**：`main.py`（10+ 方法加 ctx 参数，`self.groups: dict[str, GroupContext]`）
- **不改**：history/attribution/affinity/trigger/napcat_client/persona/scheduler 等核心模块

### 验证标准

纯重构，零行为变化。单群运行，所有可观测行为与重构前一致。

## 阶段 B：多群配置与全局串行队列

**文档**：`阶段B_多群配置与全局串行队列.md`

### 背景动机

阶段 A 搭好了 per-group 容器骨架，但实际仍是单群运行（config 只有一个 group_id，状态文件平铺）。需要实现真正的多群支持：多群配置、多群收发、多群文件隔离，以及多群 cycle 串行队列。

### 核心设计决策

- **B1**：config.yaml 群配置采用**纯列表 `group_ids: [list]`**，废弃 `group_id`，无兼容层。
- **B2**：per-group 文件存储采用**子目录方案**（`state/{group_id}/conversation.json`），非文件名后缀。手动迁移，不写迁移代码。
- **B3**：NapCatClient 多群化——`__init__` 接受 `group_ids: list[str]`，`member_cache` 嵌套化 `{group_id: {qq: info}}`，所有 API 方法接受 `group_id` 参数。单实例 + group_id 参数（非多实例）。
- **B4**：webhook 去掉 `target_group_id` 过滤，按 `data.group_id` 路由到对应 ctx，不在 groups 字典则丢弃。
- **B5**：**全局 cycle 队列**——`_cycle_queue: deque[str]` + `_cycle_queue_set: set[str]`（去重）。撞 cycle 时入队，cycle 结束后 `_drain_cycle_queue` 迭代处理（非递归）。重跑不传 soft_factors。
- **B6**：阶段 B**关闭主动触发**（`active_trigger.enabled: false`），代码保留但 `on_active_trigger` 仅改引用为 `group_ids[0]`，不实现轮询/选群。
- **B7**：NapCatClient 方法签名**直接改，不做兼容层**——一次性改完所有调用点。

### 改动范围

- **修改**：`src/config.py`（NapCatConfig 用 group_ids 替换 group_id）
- **修改**：`src/napcat_client.py`（多群化：__init__/warmup/get_nickname/所有 API 方法加 group_id）
- **修改**：`src/group_context.py`（__init__ 传入 per-group state_dir）
- **修改**：`main.py`（cycle 队列、warmup 循环创建 GroupContext、webhook 路由、_drain_cycle_queue）
- **不改**：history/attribution/affinity/trigger/persona/scheduler/llm_client 等核心模块

### 验证标准

1. 不同群被动触发正常工作
2. 大消息量下不同群维护正常回复顺序
3. 长期任务执行下维护独立的 conversation 和摘要信息

## 阶段 C：跨群印记板（已撤销）

### 历史背景

阶段 C 曾计划实现跨群感知：让 bot 在 A 群聊过的话题，到 B 群能"有印象"。方案为紧凑印记板（`CrossGroupImpressionStore`），存储每群的群印象+人印象，惰性更新（每群 trigger_count >= 5 触发 LLM 摘要），注入到 system prompt 的"# 其他群的近况"节。

### 撤销原因：DeepSeek 前缀缓存匹配效率暴跌

**部署测试发现**：阶段 C+D 上线后，API 请求的**未命中输入缓存比例从约 3.4% 飙升至 46.3%**，消费金额大幅提高。

**根因分析**：

DeepSeek 的前缀缓存机制按 messages 数组从开头逐 token 匹配，**第一个变化点之后的所有内容全部失效**。

原先利于缓存的消息结构：
```
[人格+协议+背景](system, 稳定) + [早期摘要](system, 低频变) + [近期60轮历史原文](messages, 稳定) + [当前对话](messages, 不可避免变化)
```
其中占据极大篇幅的**近期60轮历史原文完全稳定**，是前缀缓存的最大受益者，命中率极高。

阶段 C 插入 impression 后的结构：
```
[人格+协议+背景] + impression(每次变) + [早期摘要] + [近期60轮历史原文] + [当前对话]
```

`others_impressions` 几乎每次调用都在变化（不仅每5次触发更新内容，相对时间戳如"2分钟前"→"3分钟前"每次都重新计算）。impression 一变，从变化点开始 → 早期摘要 + **所有60轮历史原文** + 当前消息全部失效。

**关键认知**：任何"会变化的内容"放在 system prompt 里，都会摧毁其后所有历史消息的缓存。把 impression 调整到 summary 之后只能保住 summary（最多2000字），保不住60轮历史原文（篇幅远大于 summary），杯水车薪。

**结论**：要想保住历史原文缓存，system prompt 必须完全稳定。阶段 C 的设计与前缀缓存机制根本冲突，撤销是唯一正确选择。

### 撤销操作

通过 `git revert 3613fdd`（阶段C实施提交）完成。撤销内容：
- 删除 `src/impression.py`
- `main.py`：移除 `impression_board` 初始化、import、warmup 日志、`_run_llm_cycle` 中的 `others_impressions` 调用与 `maybe_update` 调用
- `src/persona.py`：`render_system_prompt` 恢复原签名（无 `others_impressions` 参数）、移除规则17
- `src/llm_client.py`：移除 `summarize_for_impression` 方法
- `src/history.py`：移除 `get_recent_user_contents` 方法

阶段 D 的 scheduler 改动**完整保留**，因为阶段 C 和阶段 D 改的是完全不同的代码区域，互不依赖。

### 跨群感知的未来方向

跨群感知的需求仍然存在，但实现方式需要避开前缀缓存冲突。可能的未来方向：
- 注入到 user content 而非 system prompt（user content 本来每次就变，不影响缓存）
- 作为独立的 LLM 调用（后台异步生成印记，不影响主对话的缓存）
- 这些方向均未实施，留待未来评估

## 阶段 D：主动触发多群化

**文档**：`阶段D_主动触发多群化.md`

### 背景动机

阶段 B 为了聚焦多群配置和 cycle 队列，关闭了主动触发（`enabled: false`），`on_active_trigger` 硬编码 `group_ids[0]`。现在需要补齐主动触发多群化：每群独立倒计时。

### 核心设计决策

- **D1**：调度器归属为**per-group 实例，移入 GroupContext**（方案 B），非全局单例 + per-group 状态字典。与 `_quiet_timer` 模式一致。
- **D2**：间隔计算**沿用原有随机 min-max**（`_generate_next_check_time` 不改），活跃度自适应留待未来。
- **D3**：主动触发到期时**直接 is_active=True**，不判断 pending 状态——cycle 串行机制兜底撞锁入队。
- **D4**：状态持久化为**per-group `state/{group_id}/scheduler.json`**，非全局文件。不迁移旧数据（阶段 B 期间已关闭，旧数据过时）。
- **D5**：`on_active_trigger` 改为**接受 group_id 参数**，由 per-group 调度器 Timer 直接传入。回调签名从 `Callable[[], None]` 改为 `Callable[[str], None]`。
- **D6**：**保留深夜跳过逻辑**（23:00-3:00 不触发，顺延到次日 3:00）。
- **D7**：启动初始化——**warmup 时为每群启动调度器**（遍历 self.groups），enabled=false 时不启动。

### 改动范围

- **修改**：`src/scheduler.py`（__init__ 新增 group_id 参数、start 接受 group_id、_fire 传 group_id 给回调、日志加 group_id 前缀）
- **修改**：`src/group_context.py`（__init__ 新增 `self.active_scheduler`）
- **修改**：`main.py`（删除全局 scheduler 单例、warmup 中为每群启动调度器、on_active_trigger 改为接受 group_id、run() 删除全局 scheduler 启动）
- **修改**：`config.yaml`（`active_trigger.enabled: true` 开启）

### 验证标准

1. 每群独立倒计时，`state/{group_id}/scheduler.json` 各自独立
2. 活跃群和冷清群用同一随机区间，各自独立到期
3. 深夜不触发
4. cycle 撞锁时正确入队

### 状态目录结构变化

├── {group_id}/                ├── {group_id}/
│   ├── conversation.json      │   ├── conversation.json
│   ├── state.json             │   ├── state.json
│   └── attribution_log.jsonl  │   ├── attribution_log.jsonl

## 阶段 E：多媒体素材库

### 背景动机

阶段三的 voice 双通道只支持 `ai_record` 与 `local_file`，但 `local_file` 通道长期"形同虚设"——LLM 无从知道有哪些本地文件可发；image 系统自始至终是 `EmptyImageSender` 空实现；video 完全没有支持。需要补齐多媒体发送能力，同时让 LLM 能"知道有什么可发"并按语境选择。

### 核心设计决策

- **E1：三类媒体统一双通道**——本地文件用 `data.file` 字段、网络资源用 `data.url` 字段，构造对应 OB11 段（record/image/video）。NapCat 按 OB11 schema 接受 `file`/`url` 二选一，发送器内部据此分支。
- **E2：voice 扩展为三通道**——保留 `ai_record`/`local_file`，新增 `url` 通道（`UrlVoiceSender`）。channel 字段仍作分发依据，三个 VoiceSender 各管一种。
- **E3：image/video 新增独立 sender**——`NapCatImageSender` 替换 `EmptyImageSender` 空实现；新增 `NapCatVideoSender` 与 `VideoSender` 协议接口。主流程通过接口依赖，替换零修改。
- **E4：素材库作为稳定能力注入 system prompt**——`media/manifest.json` 人工维护可用文件清单（路径/URL + 内容描述），`MediaLibrary` 启动时加载一次，渲染为 `# 你的素材库` 节，注入到 system prompt 的**稳定部分**（`# 你的过往` 之后、summary 之前）。
- **E5：LLM 直接输出路径/URL（复用 channel）**——不引入 id 间接引用。LLM 从素材库描述精确复制路径/URL 到 message 段，代码透传给 NapCat。幻觉路径由 sender 兜底（文件不存在则记 warning 跳过）。
- **E6：前缀缓存安全**——素材库**不每次 render 重读**，仅启动加载一次，行为同 background_summary。仅在 manifest 变更时断缓存一次，之后稳定，与 summary 压缩行为一致，**不重蹈阶段 C 覆辙**（阶段 C 因每次都变的相对时间戳摧毁缓存而撤销）。manifest 缺失/为空时素材库功能静默关闭，不阻塞启动。
- **E7：网络资源统一下载（media_downloader）**——所有 `data.url` 通道（image/video/voice url）先由 Bot 侧 `src/media_downloader.py` 同步下载到本地临时文件（`media/downloaded/`，sha256(url)[:16] 命名，同 URL 命中已下载文件），再转 file:// URI 发送给 NapCat。设计理念：同步下载阻塞 cycle 是合理的"真人模拟"——真人在一个群准备图片时不会跑去别的群说话，全局单队列的串行阻塞恰恰模拟了这种"专注一件事"的行为。规避 NapCat 下载 URL 资源时的 10s 内部超时硬限（AI 绘画等生成型 URL 30s+ 必定超时）。
- **E8：本地文件路径转 file:// URI**——NapCat 是独立进程，CWD 与本进程不同，相对路径无法解析（会报"识别URL失败, uri=..."）。所有本地媒体路径在 sender 内部用 `Path(path).resolve().as_uri()` 转成绝对 file:// URI 后再发送。
- **E9：sender 返回 bool + 发送失败不记录历史**——`VoiceSender`/`ImageSender`/`VideoSender` 的 `send()` 返回 `bool`（True=成功，False=失败），`main._send_messages` 检查返回值，失败时不 append 到 fast_buffer（保持历史事实性——没发出去的消息不应被 LLM 误以为已发送）。
- **E10：NapCatClient 超时参数化**——`_call` 接受 `timeout` 参数（默认 10s），`send_group_msg` 默认 30s——含媒体的 send_group_msg 需要 NapCat 上传文件到 QQ 服务器，可能超过普通 API 的 10s；纯文本消息秒级返回，不受影响。
- **E11：AI 绘画接口（sdxl_api_server.py）**——manifest 中 `type=url` 的条目可指向 SDXL 服务，GET 请求返回 `Content-Type: image/png` 图片字节流，参数用明文 query string（如 `?prompt=1girl,smile&width=1024`）。LLM 按 Booru 式标签（下划线连接单词，逗号分隔，不含空格）构造 prompt，media_downloader 下载图片字节流后转 file:// URI 发送。

### 改动范围

- **新增**：`media/manifest.json`（素材清单模板）、`src/media_library.py`（加载器）、`src/media_downloader.py`（网络资源统一下载器）、`src/senders/video_sender.py`（`NapCatVideoSender`）、`sdxl_api_server.py`（SDXL 绘画 API 服务，独立进程）、`media/downloaded/`（下载缓存目录）
- **重写**：`src/senders/image_sender.py`（`EmptyImageSender` → `NapCatImageSender`，file/url 双通道 + 下载到本地 + file:// URI 转换）
- **修改**：`src/senders/voice_sender.py`（新增 `UrlVoiceSender`，先下载到本地再走 local_file 路径）、`src/protocol.py`（新增 `VideoSender` 接口；`VoiceSender`/`ImageSender`/`VideoSender` 的 `send()` 返回 `bool`）、`src/parser.py`（`VALID_MSG_TYPES` 加入 `video`）、`src/senders/message_sender.py`（`_build_one` 透传 image/video 段）、`main.py`（初始化新 sender + 注入 timeout/temp_dir + `_send_messages` 处理 video/image 真实发送/voice-url channel + sent_ok 检查）、`src/persona.py`（加载 MediaLibrary + 注入 `# 你的素材库` + 更新输出协议示例与规则 15）、`src/napcat_client.py`（`_call` 新增 `timeout` 参数，`send_group_msg` 默认 30s）、`src/config.py`（新增 `MediaDownloadConfig`）、`config.yaml`（新增 `media_download` 配置节）

### 验证标准

1. voice `channel=url` 能先下载网络音频到本地，再走 local_file 路径发出 record 段；`channel=local_file` 行为不变
2. image 段（file/url）能真实发送，url 通道先下载到本地再转 file:// URI 发送，不再抛 `NotImplementedError`
3. video 段（file/url）能真实发送，url 通道先下载到本地再转 file:// URI 发送
4. 素材库描述正确注入 system prompt 稳定部分（`# 你的过往` 之后、summary 之前）
5. manifest 缺失/为空时 bot 正常启动，仅素材库功能关闭
6. 前缀缓存命中率不因素材库注入而持续下降（仅在 manifest 变更时短暂波动）
7. 特殊段（image/video/voice/forward）发送失败时不 append 到 fast_buffer（保持历史事实性）
8. AI 绘画接口（sdxl_api_server.py）GET 请求返回 PNG 图片字节流，media_downloader 能下载并转 file:// URI 发送
9. 本地文件路径正确转 file:// URI，不再出现"识别URL失败"错误

## 阶段 F：对话活跃度感知（direct 模式快速触发）

**文档**：`阶段F_1v1对话活跃度感知.md`

### 背景动机

低频对话场景下（尤其 1v1），Bot 回复后的后续消息往往评分不够触发"看一眼"（如"还有呢？""然后呢"等追问不命中兴趣关键词/提问语气/topic_hot），只能等 150s 静默兜底，显得迟钝。`SOFT_COOLDOWN_MIN=10s` 不是瓶颈（用户打字回复很少落在 10s 内），瓶颈是评分门槛。本阶段在不与 `topic_hot` 冲突的前提下，补齐对话快速响应能力。

### 核心设计决策

- **F1**：LLM 输出 `conversation_mode` 字段（direct=与某人往返对话 / open=开放插话），判断权归 LLM 语义判断
- **F2**：target 提取以 `targets` 字段为准（非空且可解析为有效 QQ 才进入 direct），多 target 并行（`direct_targets: dict[str, float]`，各自独立窗口）
- **F3**：新增 direct 快速触发路径，优先级介于硬因子与软因子之间，**绕过 peek_threshold**，直接让 LLM 看一眼（LLM 仍决定 silent/reply）
- **F4**：活跃窗口基于真实群聊时间戳（`last_reply_time`，避免绘图/媒体下载压缩窗口），各 target 独立 30s 衰减+续命
- **F5**：进入条件 = action=reply/multi_reply + conversation_mode=direct + targets 至少一个可解析 + Bot 实际发出回复
- **F6**：退出条件 = 自然衰减（30s 无发言）/ LLM 主动输出 open / 字典清空。**silent 保留 direct 状态**（target 可能说"哈哈"后接着追问）
- **F7**：冷却保留 SOFT_COOLDOWN_MIN=10s（避免秒回，自然对话不受影响）
- **F8**：状态不持久化（重启回落 open，不影响正确性）
- **F9**：多 target 无硬上限（靠 30s 衰减 + 10s 冷却 + LLM silent 决策三层控制）

### 改动范围

- **修改**：`src/group_context.py`（新增 `conversation_mode` + `direct_targets` 字段）

### 验证标准

1. 追问场景：用户提问→Bot 回复（direct）→用户追问→10s 内快速触发 LLM→Bot 续答
2. target 续说续命：target 30s 内连续发言，状态保持
3. 自然衰减：target 30s 无发言→从字典移除；全部过期→回落 open
4. LLM 主动退出：输出 open→清空字典
5. silent 保留 direct：silent 后 target 后续发言仍快速触发
6. 多 target 并行：Bot 回复 A 和 B 后，A/B 各自独立窗口
7. 主动开口不进入 direct：targets 为空→强制 open
8. 与 topic_hot 不冲突
9. 归因系统不受污染（direct 快速触发跳过归因）
10. 前缀缓存不下降（conversation_mode 在 assistant 输出里，不进 system prompt）

### 消息构造与前缀缓存

当前消息结构充分利用 DeepSeek 前缀缓存机制：

```
messages = [
    {"role": "system", "content": [人格+协议+背景](稳定) + [素材库](仅 manifest 变更时变) + [早期摘要](仅压缩时变)},
    *history_messages,          # 近60轮历史原文（稳定，最大篇幅，缓存主要受益者）
    {"role": "user", ...}       # 当前对话（每次变化，不可避免）
]
```

- **system prompt**：template 完全稳定；素材库与 summary 均为"低频变"内容（启动加载，不每次 render 重读），构成稳定前缀
- **history_messages**：近60轮历史原文在两次压缩之间完全稳定，是前缀缓存的最大受益者
- **new_user_content**：每次新增，不可避免要重新计算

**设计原则**：system prompt 中不放置任何高频变化的内容（这是阶段 C 撤销的核心教训）。素材库作为稳定能力启动加载一次，仅在 manifest 变更时断缓存一次，与 summary 压缩行为一致。

### 并发模型

- **接收线程**（webhook）：只写 `ctx.fast_buffer`（`_buffer_lock` 保护，持锁极短），不持任何锁
- **LLM 工作线程**：`_cycle_running` 全局标志保证串行，LLM 调用和发送不持锁
- **媒体下载**：`media_downloader.download_to_local` 在 sender 内同步调用（LLM 工作线程内），阻塞当前 cycle。设计理念：同步下载阻塞 cycle 是合理的"真人模拟"——真人在一个群准备图片时不会跑去别的群说话，全局单队列的串行阻塞恰恰模拟了这种"专注一件事"的行为
- **撞 cycle 处理**：同群注册 `ctx._cycle_pending`（cycle 内重跑），跨群入 `_cycle_queue`（cycle 后 `_drain_cycle_queue` 处理）
- **per-group 定时器**：`_quiet_timer`（150s 被动兜底）和 `active_scheduler`（随机 min-max 主动）各自独立，靠 cycle 串行机制去重

## 未来方案（已记录，当前不实现）

- **跨群感知（替代方案）**：原阶段 C 的印记板方案因前缀缓存冲突已撤销。未来如需实现跨群感知，应避开 system prompt 注入（改注入到 user content，或作为独立 LLM 调用），避免破坏前缀缓存。
- **群活跃度自适应**：根据群近期消息频率动态调整主动触发间隔，活跃的群多触发，冷清的群少触发
- **亲密度驱动的主动触发间隔**：群内亲密度高的人多时缩短间隔
- **per-group 深夜时段配置**：不同群配置不同的静默时间
- **主动触发时注入群活跃度到 prompt**：让 LLM 知道"这个群最近很活跃/冷清"（注意：需注入到 user content 而非 system prompt，避免前缀缓存冲突）
- **联网新闻获取**：主动触发时拉取热点新闻作为话题素材
- **长时间延迟回复的引用**：延迟回复跨越较长时间时自动添加引用段
- **主动触发的内容素材库**：维护 bot 的想法/槽点库，避免每次主动开口都是"突然想起来"
- **媒体下载缓存清理**：当前 `media/downloaded/` 只增不删（同 URL 命中已下载文件，轻量缓存效果）。未来如需清理，可按文件 mtime 过期或按目录大小阈值清理旧文件
